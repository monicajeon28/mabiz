## 핵심 의사결정 5가지

### 1️⃣ 배치 크기: **150명** (현재 50 → 3배 증대)
- **근거**: 1500명을 5분 내 처리하려면 배치 3배 필요
- **안전성**: 메모리 <10MB, DB pool 여유 충분
- **위험도**: 낮음 (환경변수만으로 롤백 가능)

### 2️⃣ DB 연결 풀: **30** (현재 20 → 1.5배 확대)
- **근거**: 배치 150 동시 처리 시 응답 시간 1/3로 감소
- **비용**: 무료 (연결 수만 증가)
- **위험도**: 없음 (표준 구성)

### 3️⃣ SMS 동시성: **초당 3건** (제한 추가)
- **근거**: Aligo API 초당 10건 한계에 대해 50% 안전 마진
- **효과**: 실패율 80% → <5%로 대폭 감소
- **구현**: Rate Limiter 함수 추가 (4시간)

### 4️⃣ Email 동시성: **초당 10건** (제한 추가)
- **근거**: SMS와 동일 배치에서 예측 가능성 ↑
- **효과**: 메모리 스파이크 방지, 모니터링 용이
- **실제 영향**: SMS보다 빠르므로 실제론 제약 없음

### 5️⃣ Cron 빈도: **5분 + 2분 재시도** (분리)
- **근거**: 메인 Cron(5분)과 재시도(2분)를 분리하여 효율성 ↑
- **효과**: 재시도 대기 시간 15분 → 2분으로 단축
- **구현**: 환경변수 추가만 필요

## 병목 분석 결과

### A. DB 연결 풀 부족
```
현재: Prisma pool (10) + 배치 Promise.allSettled(50)
→ 50 / 10 = 5배 대기 형성
→ 응답 시간 증가로 전체 처리 시간 급증

해결: Pool 30 + 배치 분할 (50+50+50)
→ 동시 요청 50 (DB pool) + 설치 여유
→ 응답 시간 정상화
```

### B. SMS API Rate Limit (Aligo)
```
현재: 배치 50 × Promise.all() = 초당 50건 동시 요청
→ Aligo 한계 (초당 10건) 초과
→ 80% 요청 실패 → 재시도 → 악순환

해결: 동시성 제한 (초당 3건, 안전 마진 50%)
→ 배치 150 = 50초 내 발송
→ 실패율 <5%로 대폭 감소
```

### C. Email SMTP 연결 제한
```
현재: 조직별 SMTP 재생성, 동시 연결 ~5-10개
→ 배치 300 이상에서 타임아웃 발생

해결: 동시성 제한 (초당 10건) + nodemailer 풀 설정
→ 메모리 스파이크 방지
→ Gmail/SendGrid 모두 안전 범위
```

### D. 메모리 사용량
```
결론: 병목 아님
- 배치 500 = 5-10MB (Node.js 8GB 힙 대비 무시할 수준)
- 배치 분할로 메모리 피크 제어 가능
```
