5. 기존 Contact 데이터 영향 (약 2,000,000+ 레코드)
================================================================================

현재 상태:
  - Contact.userId = null (대부분의 레코드)
  - userId가 설정된 레코드: 거의 없음 (가정: 0-1%)

FK 추가 후:
  - userId = null 레코드: 모두 유지 (ON DELETE SET NULL)
  - 문제 없음

향후 데이터 정제 (선택):
  UPDATE "Contact" c
  SET "userId" = gu."id"
  FROM "GmUser" gu
  WHERE c."userId" IS NULL
    AND c."phone" = gu."phone"
    AND gu."phone" IS NOT NULL;

  예상 영향:
  - 매칭율: 50-80% (GmUser.phone 입력률에 따라)
  - 처리 시간: 1-2분 (인덱스 활용)
  - 안전성: 높음 (phone은 고유 신뢰 데이터)

  변경:
    model Contact {
      userId     Int?
      user       GmUser?  @relation(fields: [userId], references: [id], onDelete: SetNull)
    }
    
    model GmUser {
      contacts   Contact[]
    }

  파일: prisma/migrations/[timestamp]_add_contact_gmuser_fk/migration.sql
  변경:
    ALTER TABLE "Contact" 
    ADD CONSTRAINT "Contact_userId_fkey" 
    FOREIGN KEY ("userId") REFERENCES "GmUser"("id") ON DELETE SET NULL;
