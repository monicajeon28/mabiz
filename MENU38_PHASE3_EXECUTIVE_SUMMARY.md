## 1. 핵심 이슈

### 문제 정의
Menu #38 마케팅 캠페인 기능에서 두 가지 발송 이력 모델이 병행 중:

1. **SendingHistory** (Phase 2, 기존)
   - 캠페인 발송 이력 저장
   - 메시지 내용, 메타데이터 직접 저장
   - 채널별 세분화된 상태 추적

2. **ExecutionLog** (Phase 0+2, 신규)
   - 퍼널, 자동화, 캠페인 통합 추적
   - 메시지 contentUrl (S3) 참조만
   - 월별 반복 발송 지원

### 기술적 도전
- **호환성**: ExecutionLog는 SendingHistory보다 13개 필드 부족
- **데이터 손실**: subject, body, metadata, 채널별 상태 저장 안 됨
- **응답 포맷**: 클라이언트가 기대하는 API 응답 형식과 불일치
- **성능**: Contact 조인으로 인한 응답 시간 증가

## 4. 사용자 의사결정 (선택지별 추천)

### Q1: SendingHistory API는 계속 사용할까?

**a) 계속 사용 (권장) ✅**
- 클라이언트 코드 수정 불필요 (0 비용)
- 호환성 100%
- 장기 안정성 확보
- Phase 3a 우선 진행, Phase 3b는 6개월 후

**b) 마이그레이션 (빠름)**
- 2-3주 내 완료 가능
- 클라이언트 통지 필요
- 호환성 95% (메타데이터 손실)

**c) 절충안 (복잡)**
- 두 API 모두 제공
- 클라이언트 선택
- 운영 복잡도 증가

**→ 추천: a) 계속 사용** (안정성, 비용 최적화)

---

### Q2: 메시지 본문(subject, body)은 어떻게 할까?

**현황**: API 응답에 이미 미포함 (불필요)

**a) 그냥 두기 (권장) ✅**
- 호환성 100%
- 비용 0
- 클라이언트 변화 없음

**b) contentUrl로 제공**
- S3에서 조회 가능 (느림)
- 별도 API 필요

**→ 추천: a) 그냥 두기** (불필요한 기능)

---

**현황**: API 응답에 이미 미포함 (불필요)

## 11. FAQ

### Q: SendingHistory를 계속 사용하면 DB가 무거워지지 않나?
**A**: ExecutionLog는 캠페인, 퍼널, 자동화를 모두 추적하므로 기존보다 데이터가 많습니다. 하지만 필드가 13개 적어서 저장소 효율이 높습니다. 현재 용량 내에서 1년 이상 충분합니다.

### Q: Phase 3b는 언제 해야 하나?
**A**: Phase 3a가 6개월 이상 안정적으로 운영된 후, 내부 팀이 충분히 이해한 후 진행하기를 권장합니다. 급할 필요 없습니다.

### Q: Enum 매핑에서 정보 손실이 발생하면?
**A**: INVALID_CONTACT → INVALID_PHONE 매핑은 매우 드물며, 발생 시 경고 로그가 남습니다. 실제 영향은 미미하고, 필요시 failureUserMsg로 보정할 수 있습니다.

### Q: 성능이 저하되면?
**A**: Contact 조인이 병목일 수 있습니다. 즉시 feature flag OFF로 복구하고, 캐싱/배치 쿼리로 최적화합니다. 영향 미미 (응답시간 200ms → 400ms).

### Q: 클라이언트에 알려야 하나?
**A**: Phase 3a에서는 클라이언트가 API 응답이 동일하므로 통지 불필요합니다. Phase 3b 진행 시 사전 공지 필요 (6개월 유예).
