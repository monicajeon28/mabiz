## Phase 0 — CRITICAL 보안 수정
> 우선순위: P0 | 병렬 가능: 0-1, 0-2, 0-3 동시 작업 가능

### 0-1. HTML Sanitizer 구현 (Stored XSS 차단)
- **문제**: `dangerouslySetInnerHTML`에 sanitizer 없이 사용자 HTML 렌더링
- **영향**: 크루즈닷몰 공유 DB 세션 쿠키 탈취 → 전체 시스템 장악
- **파일**:
  - `src/app/p/[slug]/LandingClient.tsx:214` — htmlContent 렌더링
  - `src/app/api/landing-pages/[id]/route.ts:36` — PATCH 저장 시점
  - `크루즈닷기능/3_B2B랜딩페이지/landing/LandingPageContent.tsx:19,23` — headerScript + htmlContent
- **작업**:
  1. `src/lib/html-sanitizer.ts` 생성 — `sanitize-html` 또는 `isomorphic-dompurify` 사용
  2. 허용 태그 화이트리스트: `<div>, <span>, <p>, <h1>-<h6>, <img>, <a>, <ul>, <ol>, <li>, <br>, <hr>, <table>, <tr>, <td>, <th>, <strong>, <em>, <b>, <i>`
  3. 차단: `<script>, <iframe>, <object>, <embed>, on* 이벤트 핸들러, javascript: URI`
  4. `headerScript` 필드 → GTM/Pixel ID만 허용하는 화이트리스트 방식으로 전환 (자유 JS 실행 금지)
  5. PATCH API 저장 시점에서 sanitize 적용
  6. 공개 렌더링(`LandingClient.tsx`)에서도 2차 sanitize 적용

### 0-2. Origin Guard 수정 + Rate Limiting
- **문제**: Origin 없으면 통과, 환경변수 없으면 모든 Origin 허용
- **파일**:
  - `src/lib/origin-guard.ts:11,19`
- **작업**:
  1. `origin-guard.ts:11` → `if (!origin) return false` (Origin 없는 요청 차단)
  2. `origin-guard.ts:19` → `if (allowedOrigins.length === 0) return false` (환경변수 없으면 차단)
  3. Vercel Edge Middleware 또는 Upstash Ratelimit으로 IP 기반 rate limit 구현:
     - `/api/landing-pages/*/register`: 분당 3회
     - `/api/landing-pages/*/view`: 분당 10회
     - `/api/public/landing/*/comments`: 분당 5회
  4. 서버 사이드 CAPTCHA(Cloudflare Turnstile) 검토 (register 엔드포인트)

### 0-3. 웹훅 processedWebhookEvent 트랜잭션 통합
- **문제**: processedWebhookEvent 기록이 비즈니스 로직 트랜잭션 밖 → 중복 처리 가능
- **파일**: 모든 웹훅 라우트 (`src/app/api/webhooks/*/route.ts`)
- **작업**:
  1. reservation 웹훅 패턴을 표준으로 채택 (processedWebhookEvent를 $transaction 내부에 포함)
  2. purchase, refund, inquiry, gold-inquiry, partner-signup, news-sync 6개 웹훅에 동일 패턴 적용
  3. 웹훅 응답에서 `contactId`, `leadId` 등 내부 DB ID 제거

  2. VipCareSequence 완료/실패 시 ContactFunnelState도 동기적으로 전이

### 1-3. 인덱스 5건 + FK 5건 추가
- **문제**: 풀스캔 쿼리, FK 누락으로 고아 레코드 발생
- **파일**: `prisma/schema.prisma`
- **작업** (수동 SQL 마이그레이션, `CREATE INDEX CONCURRENTLY` 사용):
  ```sql
  -- 인덱스 추가
  CREATE INDEX CONCURRENTLY idx_vipcare_contact_funnel_status 
    ON "VipCareSequence"("contactId", "funnelId", "status");
  CREATE INDEX CONCURRENTLY idx_landing_reg_phone 
    ON "CrmLandingRegistration"("phone");
  CREATE INDEX CONCURRENTLY idx_vipcarelog_seq_status_scheduled 
    ON "VipCareLog"("sequenceId", "status", "scheduledAt");
  CREATE INDEX CONCURRENTLY idx_smslog_sentat 
    ON "SmsLog"("sentAt");
  CREATE INDEX CONCURRENTLY idx_funnel_state_org_status_scheduled 
    ON "ContactFunnelState"("organizationId", "status", "nextScheduledAt");
  ```
  - FK 추가: CrmLandingPage.groupId → ContactGroup (onDelete: SetNull)
  - FK 추가: CrmLandingPage.autoFunnelId → Funnel (onDelete: SetNull)
  - PayAppPayment에 organizationId 컬럼 추가 (테넌트 격리)

> **주의**: 모든 인덱스/FK 추가는 수동 SQL로 `CONCURRENTLY` 사용. `prisma migrate deploy` 금지.
