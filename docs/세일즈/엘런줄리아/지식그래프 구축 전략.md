## Part 1: 아키텍처 개요 및 소스 마이닝 전략 (Architecture & Mining)

이 시스템의 대전제는 **"코드의 품질은 학습한 패턴의 품질을 넘을 수 없다"**는 것입니다. 따라서 Phase 1-5는 단순한 데이터 수집이 아니라, **글로벌 엔지니어링 정수(Essence)를 추출하는 과정**입니다.

### 1.1 전체 시스템 아키텍처의 의도 (Executive Intent)

이 시스템은 '코드 자동 완성'을 넘어선 **'아키텍처 제안 시스템'**입니다.

- **AS-IS (기존 LLM):** "코드를 짜줘"라고 하면 확률적으로 가장 그럴듯한 텍스트를 생성. 문맥이 길어지면 할루시네이션 발생.
    
- **TO-BE (Knowledge Graph System):**
    
    1. 검증된 12,960개의 패턴 중 현재 상황에 맞는 최적의 설계를 검색(Retrieval).
        
    2. 검색된 **구조(Structure)**와 **규칙(Rule)**을 LLM에 주입.
        
    3. LLM은 이 가이드라인 안에서 코드를 생성.
        

**[데이터 흐름의 대전략]**

```

                                               │

                                      [Phase 7-10: RAG & Automation] (실시간 활용)
```

#### **1.2.1 선정 기준의 역공학 (Reverse Engineering Criteria)**

명세서에 언급된 프로젝트(`spark`, `vscode`, `tokio`, `express`)를 분석해보면 다음과 같은 선정 기준이 적용되었음을 알 수 있습니다.

|**기준 (Criteria)**|**전략적 의도 (Strategic Intent)**|**대표 프로젝트**|

|**High Complexity**|단순한 CRUD가 아닌, 복잡한 동시성/비동기 처리가 완벽하게 구현된 프로젝트.|`apache/spark` (분산처리), `tokio-rs/tokio` (비동기 런타임)|

### 1.3 패턴 추출 로직의 심층 기술 분석 (Pattern Extraction Logic)

데이터 파이프라인의 핵심 엔진인 `PatternExtractor` 클래스를 줄 단위 로직으로 분석합니다. 이것은 **텍스트 처리(NLP)**가 아닌 **컴파일러 기술(AST)**의 영역입니다.

#### **1.3.1 분류 체계 (Taxonomy) 설계**

명세서의 `PATTERN_TYPES` 딕셔너리는 시스템이 이해해야 할 지식의 범위를 정의합니다.

Python

```
PATTERN_TYPES = {
    # 1. 고전적 OOP 패턴 (GoF)
    "creational": ["Factory", "Builder", ...],
    
    # 2. 현대적 시스템 아키텍처
    "architectural": ["Clean Architecture", "Hexagonal"],
    
    # 3. 최우선 순위 (High-Leverage) 기술
    "async_pattern": ["Async/Await", "Event Loop"],
    "concurrency": ["Thread Pool", "Lock", "Channel"],
    
    # 4. 언어 특화 숙련도
    "python_idioms": ["Context Manager", "Decorator"]
}
```

- **전략적 중요성:** 단순한 디자인 패턴뿐만 아니라 **`async_pattern`과 `concurrency`**가 포함된 것이 핵심입니다. 이는 초보 개발자가 가장 많이 실수하고, LLM이 가장 자주 틀리는 '난제' 영역이기 때문입니다. 이를 명시적으로 추출함으로써 시스템의 가치를 극대화합니다.
    

#### **1.3.2 추출 프로세스 (The Mining Flow)**

`extract_from_repo` 메서드의 내부 동작을 시뮬레이션합니다.

1. **Clone (물리적 확보):**
    
    - `git clone --depth 1` (최신 버전만 빠르게 확보)을 수행하여 로컬 파일시스템에 소스를 가져옵니다.
        
2. **Walk & AST Parse (구조적 해석):**
    
    - 단순히 파일을 읽는 것(`read()`)이 아니라, 언어별 파서(Python의 `ast` 모듈, TS의 `typescript-compiler` 등)를 사용하여 코드를 **추상 구문 트리(AST)**로 변환합니다.
        
    - **왜 Regex(정규식)가 아닌가?**
        
        - Regex는 주석에 있는 "Factory" 단어와 실제 클래스 이름 `UserFactory`를 구분하기 어렵습니다.
            
        - AST는 `ClassDef` 노드 밑에 `FunctionDef`가 있고, 그 이름이 `create_`로 시작하며 반환 타입이 추상 클래스인 구조를 정확히 파악합니다.
            
3. **Detect (패턴 매칭):**
    
    - 정의된 구조적 시그니처와 AST를 대조합니다.
        
    - _Example:_ Python의 `Context Manager` 패턴 감지
        
        - 조건: 클래스 내부에 `__enter__`와 `__exit__` 메서드가 모두 존재하는가? -> **True** -> 추출.
            
4. **Normalize (데이터 정규화):**
    
    - 추출된 정보를 통일된 형식(JSON/Dict)으로 변환합니다. 이때 코드의 시작 라인(`line_start`)과 끝 라인(`line_end`)을 기록하여 나중에 원본 참조가 가능하게 합니다.

## Part 2: AST 기반 패턴 추출 및 정규화 (AST Analysis)

시스템의 품질은 **"얼마나 정확하게 패턴을 인지했는가"**에 달려 있습니다. 단순히 파일 내에 'Factory'라는 단어가 있다고 해서 Factory 패턴이 아닙니다. 이 시스템은 **AST(Abstract Syntax Tree, 추상 구문 트리)**를 통해 코드의 '의도'와 '구조'를 파악합니다.

### 2.1 왜 AST인가? (The Engineering "Why")

텍스트 검색(Regex)과 구조 분석(AST)의 결정적 차이를 이해해야 합니다.

#### **시나리오: Singleton 패턴 감지**

- **Case A: 텍스트 검색 (Regex) 실패 사례**
    
    Python
    
    ```
    # TODO: This class behaves like a Singleton but isn't one.
    class UserConfig:
        pass
    ```
    
    - Regex는 주석(`Match "Singleton"`)을 보고 이를 패턴으로 오인할 수 있습니다. (False Positive)
        
- **Case B: AST 분석 성공 사례**
    
    - AST는 주석을 무시하고 코드의 **논리 구조**만 봅니다.
        
    - **탐지 로직:**
        
        1. `ClassDef` 노드 발견 (`UserConfig`)
            
        2. 내부에 `_instance`라는 스태틱 변수가 있는가? -> **No**
            
        3. `__new__` 메서드에서 인스턴스 반환 로직이 있는가? -> **No**
            
        4. **판정:** Singleton 패턴이 아님.
            

이처럼 AST는 코드를 **트리 구조의 데이터**로 변환하여 부모-자식 관계, 스코프, 타입 등을 정확하게 파악합니다.

### 2.2 AST 파싱 파이프라인 (The Parsing Pipeline)

명세서의 `extract_from_repo` 내부에서는 각 언어별로 특화된 **파서(Parser)**가 동작합니다. 이를 3단계로 분해합니다.

```
┌─────────────────┐      ┌─────────────────┐      ┌─────────────────┐
│  Source Code    │      │  Lexical Analysis│      │  Syntax Analysis│
│  (String)       │ ──▶  │  (Tokens)       │ ──▶  │  (AST Nodes)    │
│  "def func()..."│      │  [DEF, ID, (..)]│      │  FunctionDef(..)│
└─────────────────┘      └─────────────────┘      └─────────────────┘
```

#### **2.2.1 언어별 파서 매핑 (Polyglot Support)**

이 시스템은 다국어를 지원하므로, 각 언어에 맞는 파싱 엔진을 내장하고 있습니다.

|**언어**|**파싱 라이브러리**|**역할**|
|---|---|---|
|**Python**|`ast` (Standard Lib)|파이썬 코드를 파이썬 객체 트리로 변환.|
|**TypeScript**|`typescript` Compiler API|타입 정보까지 포함된 깊은 구조 분석 가능.|
|**JavaScript**|`acorn` / `babel-parser`|최신 ES6+ 문법(Arrow func 등) 지원.|
|**Rust**|`syn` Crate|Rust의 매크로 및 소유권 구조 파싱.|

### 2.3 패턴 감지 알고리즘 상세 (Detection Logic)

`PatternExtractor`가 실제로 어떻게 패턴을 찾아내는지, **Visitor 패턴**을 적용한 Python 의사 코드(Pseudo-code)로 구현 로직을 상세화합니다.

#### **예시: Python의 'Context Manager' 패턴 추출**

Context Manager는 `with` 구문을 사용할 수 있게 해주는 패턴으로, 리소스 관리(파일 닫기, DB 연결 해제 등)에 필수적입니다.

Python

```
import ast

class PatternVisitor(ast.NodeVisitor):
    def __init__(self):
        self.patterns = []

    def visit_ClassDef(self, node):
        """클래스 정의 노드를 방문할 때 실행"""
        
        # 1. 메서드 이름 추출
        method_names = [m.name for m in node.body if isinstance(m, ast.FunctionDef)]
        
        # 2. 시그니처 매칭: __enter__ 와 __exit__가 모두 존재하는가?
        has_enter = "__enter__" in method_names
        has_exit = "__exit__" in method_names
        
        if has_enter and has_exit:
            # 3. 패턴 확정 및 데이터 추출
            self.patterns.append({
                "type": "Context Manager",
                "name": node.name,
                "line_start": node.lineno,
                "line_end": node.end_lineno,
                "code": self._get_source_segment(node), # 소스 코드 잘라내기
                "docstring": ast.get_docstring(node)
            })
        
        # 자식 노드 계속 순회
        self.generic_visit(node)

# 실행
tree = ast.parse(source_code)
visitor = PatternVisitor()
visitor.visit(tree)
```

**[핵심 분석]**

- `visit_ClassDef`: 트리를 순회하다가 '클래스 정의'를 만나면 멈춥니다.
    
- `method_names`: 클래스 내부의 함수 정의(`FunctionDef`)만 골라내어 이름을 리스트업합니다.
    
- **Logical Check (`and`):** `__enter__`와 `__exit__`가 **동시에** 있어야만 Context Manager로 인정합니다. 이것이 AST의 강력함입니다.
    

### 2.4 데이터 정규화 (Normalization)

추출된 AST 노드는 Raw Data입니다. 이를 Neo4j에 넣기 좋게, 그리고 RAG 검색 시 LLM이 이해하기 좋게 **정제(Cleaning)하고 표준화(Standardization)**하는 과정이 필수적입니다.

#### **2.4.1 메타데이터 표준화 (Schema Mapping)**

서로 다른 언어에서 추출된 패턴을 하나의 통일된 JSON 스키마로 변환합니다.

- **Input (Raw):**
    
    - Python: `ClassDef`, `FunctionDef`
        
    - JS: `ClassDeclaration`, `ArrowFunctionExpression`
        
- **Output (Normalized Pattern Object):**
    
    JSON
    
    ```
    {
      "pattern_id": "uuid_v4",
      "name": "DatabaseConnector",
      "category": "python_idioms",  // 매핑 테이블에 의해 결정
      "pattern_type": "Context Manager",
      "language": "python",
      "source_location": {
        "file": "src/db/connector.py",
        "start": 45,
        "end": 82
      },
      "complexity_score": 12, // 순환 복잡도(Cyclomatic Complexity) 계산
      "dependencies": ["sqlalchemy", "contextlib"] // import 문 분석 결과
    }
    ```
    

#### **2.4.2 코드 정제 (Sanitization)**

임베딩 비용 절감과 검색 정확도를 위해 코드를 다듬습니다.

1. **주석 제거/요약:** 불필요한 주석(Copyright 헤더 등)을 제거하고, Docstring은 `description` 필드로 이동시킵니다.
    
2. **공백 정규화:** 탭(Tab)을 스페이스(Space)로 변환하고, 불필요한 빈 줄을 제거하여 토큰 수를 최적화합니다.
    

### 2.5 엣지 케이스 처리 (Edge Cases)

이 단계에서 시스템의 견고함(Robustness)이 결정됩니다.

- **중첩 클래스 (Nested Classes):** 클래스 안에 클래스가 있는 경우, 부모-자식 관계를 유지하며 재귀적으로 추출합니다.
    
- **비완전 코드 (Incomplete Code):** 파싱 에러(`SyntaxError`)가 발생하는 파일은 로그(`mining_errors.log`)를 남기고 스킵하여 전체 파이프라인이 멈추지 않도록 처리합니다.
    
- **너무 긴 파일:** 10,000 라인이 넘는 파일은 모듈 단위로 쪼개서 분석하거나, 메모리 보호를 위해 스킵하는 임계값(Threshold)을 설정합니다.

## Part 3: 지식그래프 스키마 엔지니어링 (Graph Schema Design)

데이터베이스 스키마는 시스템의 세계관을 결정합니다. 관계형 데이터베이스(RDB)가 '엑셀 표'라면, 지식그래프(Graph DB)는 '마인드맵'입니다. AI가 "이 에러는 저 패턴과 관련이 있고, 그 패턴은 이 프로젝트에서 유래했다"라고 추론할 수 있는 것은 바로 이 스키마 덕분입니다.

### 3.1 모델링 전략: 왜 그래프인가? (The Modeling Strategy)

이 시스템이 RDB(PostgreSQL 등) 대신 Neo4j를 선택한 기술적 이유는 **'조인(Join) 없는 탐색'** 때문입니다.

- **RDB의 한계:** "비동기(Async) 카테고리의 패턴 중, Python 언어이고, 에러 처리(Error Handling)와 관련된 것을 찾아라"라는 쿼리를 날리면 거대한 테이블들을 조인해야 하므로 느리고 복잡합니다.
    
- **Graph의 이점:** `(:Pattern)-[:BELONGS_TO]->(:Category)`와 같이 물리적으로 연결되어 있어, 연결선을 따라가기만 하면 됩니다(Index-free Adjacency). 이는 실시간 RAG 검색 속도를 획기적으로 높입니다.
    

### 3.2 노드(Node) 설계 명세

노드는 그래프의 '점'이자 데이터의 실체입니다. 각 노드 타입(Label)별 속성과 설계 의도를 분석합니다.

#### **(1) Pattern 노드 (The Core Atom)**

시스템의 주인공입니다. Part 2에서 정규화된 데이터가 이 노드로 변환됩니다.

|**속성 (Property)**|**데이터 타입**|**설명 및 설계 의도**|
|---|---|---|
|`id`|String (UUID)|고유 식별자. 업데이트/삭제 시 무결성 보장.|
|`name`|String|패턴 이름 (예: "Async Context Manager").|
|`embedding`|List<Float>|**[핵심]** 1536차원 벡터. 시맨틱 검색의 대상.|
|`problem`|String|해결하려는 문제 상황. 임베딩의 주 재료.|
|`solution`|String|해결 방법 설명.|
|`rule`|String|적용 규칙 (강제성 부여).|
|`code_chunk`|String|실제 코드 블록 (토큰 최적화됨).|
|`language`|String|프로그래밍 언어 (필터링 속도 최적화를 위해 속성으로 저장).|

#### **(2) Repository 노드 (The Source of Truth)**

패턴의 출처입니다. 신뢰성(Trust)을 담당합니다.

|**속성**|**데이터 타입**|**설명**|
|---|---|---|
|`url`|String|GitHub URL.|
|`stars`|Integer|프로젝트의 중요도 가중치(Weight)로 사용됨.|
|`last_analyzed`|DateTime|데이터 최신성 관리용.|

#### **(3) Error 노드 (The Problem Space)**

나중에 Part 7(에러 로깅)에서 다루겠지만, 스키마에는 미리 정의되어야 합니다.

- **역할:** "이 패턴은 어떤 에러를 해결했는가?"를 기록.
    
- **속성:** `error_message`, `stack_trace`, `embedding`.
    

### 3.3 관계(Relationship) 엔지니어링

그래프의 꽃은 '선(Edge)'입니다. 관계를 어떻게 정의하느냐가 AI의 추론 능력을 결정합니다.

Cypher

```
// 핵심 관계도 (Conceptual Schema)
(:Pattern)-[:EXTRACTED_FROM]->(:Repository)
(:Pattern)-[:SOLVES]->(:Error)
(:Pattern)-[:RELATED_TO {weight: 0.8}]->(:Pattern)
(:Pattern)-[:BELONGS_TO]->(:Category)
```

#### **3.3.1 `EXTRACTED_FROM` (출처 추적)**

- **방향:** `Pattern` -> `Repository`
    
- **의미:** 계보(Lineage) 관리.
    
- **활용:** "Microsoft VSCode 팀은 이 문제를 어떻게 해결했나?" 같은 질의 시 특정 레포지토리의 패턴만 필터링하여 검색.
    

#### **3.3.2 `RELATED_TO` (연관 추천)**

- **방향:** 양방향 (Undirected)
    
- **속성:** `weight` (0.0 ~ 1.0)
    
- **생성 로직:**
    
    - 두 패턴이 같은 파일 내에 인접해 있었거나,
        
    - 같은 함수 내에서 호출되었거나,
        
    - 임베딩 유사도가 높을 때 자동으로 연결.
        
- **전략적 가치:** 사용자가 A 패턴을 검색했을 때, "이걸 쓰려면 B 패턴도 필요할걸요?"라고 제안(Suggestion)하는 기반이 됩니다.
    

### 3.4 Cypher 구현 코드 (Implementation)

실제 Neo4j 데이터베이스에 이 스키마를 적용하는 **DDL(Data Definition Language)** 쿼리입니다. 성능과 무결성을 위한 제약조건(Constraint) 설정이 핵심입니다.

Cypher

```
// 1. Uniqueness Constraints (중복 방지 및 빠른 조회)
CREATE CONSTRAINT pattern_id_unique IF NOT EXISTS
FOR (p:Pattern) REQUIRE p.id IS UNIQUE;

CREATE CONSTRAINT repo_url_unique IF NOT EXISTS
FOR (r:Repository) REQUIRE r.url IS UNIQUE;

// 2. Indexes (검색 성능 최적화)
// 이름, 언어, 카테고리는 정확히 일치하는 검색(Exact Match)이 빈번하므로 B-Tree 인덱스 생성
CREATE INDEX pattern_name_idx IF NOT EXISTS FOR (p:Pattern) ON (p.name);
CREATE INDEX pattern_lang_idx IF NOT EXISTS FOR (p:Pattern) ON (p.language);
CREATE INDEX pattern_cat_idx IF NOT EXISTS FOR (p:Pattern) ON (p.category);

// 3. Fulltext Index (키워드 검색용 - 하이브리드 검색 지원)
CREATE FULLTEXT INDEX pattern_text_idx IF NOT EXISTS
FOR (p:Pattern) ON EACH [p.name, p.problem, p.description];
```

> **Strategist Note:** 여기서 `Vector Index` 생성 구문은 Part 5(저장소 최적화)에서 다룹니다. 지금은 구조적 인덱스(B-Tree, Fulltext)에 집중합니다.

### 3.5 모델링 최적화 전략 (Denormalization Decision)

데이터베이스 설계 시 항상 고민하는 **정규화(Normalization) vs 비정규화(Denormalization)** 이슈가 그래프에서도 발생합니다.

- **고민:** `Language`를 별도 노드 `(:Language {name: "python"})`로 뺄 것인가, 아니면 Pattern 노드의 속성 `p.language = "python"`으로 넣을 것인가?
    
- **결정:** **속성(Property)으로 저장 (비정규화)**
    
- **이유:**
    
    - RAG 검색 시 "Python 언어 내에서 검색해줘"라는 요청은 필터링(Filtering) 조건입니다.
        
    - 그래프 탐색(Traverse)보다는 인덱스 조회(Index Lookup)가 훨씬 빠릅니다.
        
    - 만약 `(:Language)` 노드로 만들면, Python 노드 하나에 4,000개의 패턴이 연결되는 **Super Node(Dense Node)** 문제가 발생하여 성능이 저하될 위험이 있습니다.

## Part 4: 임베딩 파이프라인 아키텍처 (Embedding Pipeline)

임베딩(Embedding)은 자연어를 고차원의 좌표값(Vector)으로 변환하는 과정입니다. 이 시스템은 12,960개의 패턴을 1536차원의 숫자로 변환합니다. 여기서 엔지니어링의 핵심 목표는 **품질(Quality), 비용(Cost), 속도(Speed)**의 균형을 맞추는 것입니다.

### 4.1 파이프라인 아키텍처 (The Flow)

단일 스레드로 하나씩 API를 호출하는 것은 비효율적입니다. 우리는 **생산자-소비자 패턴(Producer-Consumer Pattern)**에 기반한 배치 파이프라인을 구축합니다.

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  Source Reader  │    │  Batch Builder  │    │  Async Request  │    │  Vector Writer  │
│  (Neo4j Fetch)  │───▶│  (Aggregator)   │───▶│  (OpenAI API)   │───▶│  (Neo4j Update) │
│  "Stream Data"  │    │  [P1, P2...P50] │    │  "Parallel"     │    │  "Bulk Write"   │
└─────────────────┘    └─────────────────┘    └─────────────────┘    └─────────────────┘
         ▲                                                                     │
         └───────────────────────── Checkpoint ────────────────────────────────┘
```

- **Source Reader:** Neo4j에서 아직 임베딩되지 않은(`embedding IS NULL`) 패턴만 스트리밍으로 가져옵니다. (멱등성 보장)
    
- **Batch Builder:** API 호출 횟수를 줄이기 위해 데이터를 50~100개 단위로 묶습니다.
    
- **Async Request:** 네트워크 대기 시간(Latency)을 최소화하기 위해 비동기 I/O를 사용합니다.
    
- **Vector Writer:** 변환된 벡터값을 다시 DB에 업데이트합니다.
    

### 4.2 임베딩 텍스트 설계 (Text Engineering)

가장 중요한 기술적 결정입니다. "무엇을 벡터로 만들 것인가?"

소스 코드 전체를 통째로 임베딩하면 노이즈가 너무 많아 검색 정확도가 떨어집니다. 따라서 **'의미론적 요약본(Semantic Summary)'**을 동적으로 생성해야 합니다.

#### **4.2.1 필드 조합 전략 (Concatenation Strategy)**

`prepare_embedding_text` 함수는 각 패턴 객체에서 가장 "검색에 유용한" 정보만 추출하여 하나의 문자열로 합칩니다.

Python

```
def prepare_embedding_text(pattern: dict) -> str:
    # 1. 메타데이터: 검색의 '방향'을 결정
    meta = f"Language: {pattern['language']}\nCategory: {pattern['category']}"
    
    # 2. 문제와 해결: 검색의 '핵심 의도' (가중치 높음)
    context = f"Problem: {pattern['problem']}\nSolution: {pattern['solution']}"
    
    # 3. 코드: 구체적 구현 (토큰 제한 적용)
    # 코드가 너무 길면 의미가 희석되므로 앞부분 500자(약 150토큰)만 사용
    code_snippet = f"Code:\n{pattern['code_example'][:500]}"
    
    # 최종 결합
    return f"{meta}\n\n{context}\n\n{code_snippet}"
```

- **전략적 이유:**
    
    - `Language`를 텍스트에 포함하면, 벡터 공간상에서 Python 관련 패턴들은 서로 가깝게 모이게 됩니다(Clustering 효과).
        
    - `Code`를 자르는(Truncation) 이유는 임베딩 모델의 입력 한도(8191 토큰) 문제보다는, **'핵심 로직'에 집중하기 위함**입니다. 아키텍처 패턴의 정체성은 보통 클래스 정의부와 초기 메서드에 드러납니다.
        

### 4.3 모델 선정 및 API 최적화

#### **4.3.1 모델 선정: `text-embedding-3-small`**

- **Small vs Large:**
    
    - `large` (3072차원): 미세한 뉘앙스 구분에 유리하지만 비용이 비싸고 저장 공간이 2배 필요.
        
    - `small` (1536차원): 코드 패턴 검색에는 충분한 성능(MTEB 벤치마크 기준)을 보여주며, **비용이 1/5 수준**으로 저렴함. 속도가 훨씬 빠름.
        
    - **결정:** 대량의 패턴을 처리해야 하므로 **Small 모델**을 선택하여 가성비와 속도를 최적화.
        

#### **4.3.2 배치 처리와 Rate Limiting**

OpenAI API는 분당 요청 제한(RPM, TPM)이 있습니다. 이를 위반하지 않으면서 처리량을 극대화해야 합니다.

- **Batch Size:** 50
    
    - 너무 작으면: 네트워크 오버헤드 증가.
        
    - 너무 크면: 하나의 긴 텍스트 때문에 전체 배치가 실패하거나 토큰 한도 초과 위험.
        
    - 50개씩 묶으면 한 번 호출로 50개의 벡터를 얻으므로 **API 호출 횟수가 1/50로 감소**.
        
- **Rate Limiter 구현:**
    
    Python
    
    ```
    import time
    from tenacity import retry, wait_exponential, stop_after_attempt
    
    @retry(wait=wait_exponential(multiplier=1, min=4, max=10), stop=stop_after_attempt(5))
    def get_embeddings_with_retry(texts):
        # 429 Too Many Requests 에러 발생 시 지수 백오프(Exponential Backoff)로 대기 후 재시도
        return client.embeddings.create(input=texts, model="text-embedding-3-small")
    ```
    
    - 단순 `sleep`이 아닌 **지수 백오프(Exponential Backoff)**를 사용하여 서버 부하에 따라 유동적으로 대기 시간을 조절합니다.
        

### 4.4 장애 대응: 체크포인트 시스템 (Checkpointing)

12,000개를 처리하다가 11,000번째에서 네트워크 오류로 스크립트가 죽으면? 처음부터 다시 하는 것은 시간과 돈의 낭비입니다.

- **상태 저장:** 처리가 완료된 패턴에는 Neo4j에 `embedded_at` 타임스탬프를 찍거나, 별도의 `state.json` 파일에 마지막 처리한 Cursor ID를 기록합니다.
    
- **재시작 전략:** 파이프라인이 다시 시작될 때, `WHERE p.embedding IS NULL` 조건으로 쿼리하여 **안 된 것만 골라서** 처리합니다.
    

### 4.5 비용 분석 (Cost Projection)

엔지니어링에서 예산 산정은 필수입니다.

- **총 패턴 수:** 12,960개
    
- **패턴당 평균 토큰:** 약 150 토큰 (텍스트 요약 + 코드 일부)
    
- **총 토큰:** $12,960 \times 150 \approx 1,944,000$ 토큰 (약 200만 토큰)
    
- **가격:** $0.00002 / 1K tokens
    
- **총 비용:** $1,944 \times 0.00002 \approx \$0.04$
    

> **Strategist Note:** 예상보다 비용이 매우 저렴합니다(약 50원). 이는 `text-embedding-3-small` 모델의 극적인 효율성 때문입니다. 따라서 비용 걱정보다는 **데이터의 품질과 파이프라인의 안정성**에 투자를 집중해도 된다는 전략적 결론을 내릴 수 있습니다.

### 4.6 벡터 저장 (Storage)

API에서 받은 `List<Float>` 형태의 1536개 숫자를 Neo4j에 저장합니다.

Cypher

```
// Cypher: 배치 업데이트 쿼리
UNWIND $batch_data AS row
MATCH (p:Pattern {id: row.id})
SET p.embedding = row.vector,
    p.embedded_at = datetime()
```

- `UNWIND`: 50개의 데이터를 한 번의 DB 트랜잭션으로 처리하여 DB 부하를 줄입니다.

## Part 5: 벡터 인덱싱 및 저장소 최적화 (Vector Storage)

임베딩된 벡터 데이터($12,960 \times 1536$ Matrix)는 거대한 숫자 덩어리입니다. 인덱스 없이 검색하면 시스템은 모든 패턴을 하나씩 비교하는 **Full Scan ($O(N)$)**을 수행해야 하며, 이는 사용자 경험을 망치는 주범이 됩니다. 우리는 **Approximate Nearest Neighbor (ANN)** 검색을 통해 이를 **$O(\log N)$** 수준으로 최적화합니다.

### 5.1 Neo4j Vector Index의 구조 (The Mechanism)

Neo4j 5.x 버전부터 도입된 **Vector Index**는 루씬(Lucene) 기반의 벡터 검색 엔진을 내장하고 있습니다.

#### **5.1.1 인덱스 생성 매커니즘**

명세서에 있는 Cypher 쿼리의 각 옵션이 기술적으로 어떤 의미인지 분석합니다.

Cypher

```
CREATE VECTOR INDEX pattern_embedding_idx IF NOT EXISTS
FOR (p:Pattern) ON (p.embedding)
OPTIONS {
    indexConfig: {
        `vector.dimensions`: 1536,
        `vector.similarity_function`: 'cosine'
    }
}
```

1. **`FOR (p:Pattern)`:** 특정 레이블에만 인덱스를 적용하여 불필요한 연산을 방지합니다.
    
2. **`vector.dimensions: 1536`:** 차원 수가 맞지 않는 벡터가 들어오면 인덱싱에서 제외하거나 에러를 발생시켜 데이터 무결성을 지킵니다.
    
3. **`vector.similarity_function: 'cosine'`:**
    
    - **Euclidean (L2):** 두 점 사이의 '거리'를 측정. 텍스트 길이가 다르면 거리가 멀어질 수 있음.
        
    - **Cosine:** 두 벡터 사이의 '각도'를 측정. 텍스트 길이(Magnitude)가 달라도 **주제(Direction)가 같으면 유사하다고 판단.** 코드 패턴 검색에는 Cosine이 압도적으로 유리합니다.
        

### 5.2 내부 알고리즘: HNSW (Hierarchical Navigable Small World)

Neo4j 벡터 인덱스는 내부적으로 **HNSW 알고리즘**을 사용합니다. 이것이 빠른 검색의 비결입니다. 이 알고리즘을 이해해야 튜닝이 가능합니다.

- **구조 (The Layers):** 데이터 포인트들을 여러 층(Layer)의 그래프로 구성합니다.
    
    - **Top Layer (고속도로):** 데이터들이 듬성듬성 연결되어 있습니다. 여기서 쿼리 벡터와 가장 가까운 진입점을 찾아 크게 점프합니다.
        
    - **Lower Layers (골목길):** 아래층으로 내려갈수록 연결이 촘촘해집니다. 점점 더 정밀하게 가장 가까운 이웃을 찾아 좁혀갑니다.
        
- **작동 원리:** 마치 고속도로를 타고 근처 도시로 간 뒤(Top), 국도로 동네에 진입하고(Middle), 골목길로 집을 찾는(Bottom) 방식과 같습니다. 이를 통해 수만 개의 데이터를 다 뒤지지 않고도 목적지를 찾습니다.
    

### 5.3 인덱스 성능 튜닝 및 최적화 (Performance Tuning)

명세서에는 기본 설정만 나와 있지만, 대규모 운영 시 고려해야 할 히든 파라미터들이 있습니다.

#### **5.3.1 `m` (Connections per Node)**

- 노드 하나당 연결할 이웃의 최대 개수입니다.
    
- **높으면:** 검색 정확도가 올라가지만, 인덱스 크기가 커지고 생성 시간이 오래 걸립니다.
    
- **낮으면:** 메모리를 적게 쓰지만, '고립된 섬'이 생겨 검색 결과가 부정확해질 수 있습니다.
    
- **전략:** 코드 패턴 데이터는 12,960개로 아주 많지 않으므로, 기본값(보통 16~32)보다 약간 높게 설정하여 **정확도(Recall)를 최우선**으로 합니다.
    

#### **5.3.2 `ef_construction` & `ef_search`**

- **`ef_construction`:** 인덱스를 _만들 때_ 얼마나 많은 후보를 탐색할지. 높을수록 인덱스 품질이 좋아집니다. (빌드 시간은 증가)
    
- **`ef_search`:** _검색할 때_ 얼마나 많은 후보를 방문할지. 높을수록 정확하지만 느려집니다.
    
- **최적화 전략:** RAG 시스템은 0.1초의 차이보다 '정확한 솔루션'이 중요하므로, 쿼리 시 `ef_search` 값을 넉넉하게 주어 **정확도를 확보**합니다.
    

### 5.4 인덱스 상태 관리 및 모니터링

인덱스는 생물처럼 관리되어야 합니다. 데이터가 추가될 때마다 백그라운드에서 갱신되기 때문입니다.

Cypher

```
// 인덱스 상태 정밀 점검
SHOW INDEXES YIELD name, type, state, populationPercent, uniqueness
WHERE name = 'pattern_embedding_idx'
```

- **`state`:** `ONLINE`이어야 정상 검색 가능. `POPULATING`이면 아직 빌드 중입니다.
    
- **`populationPercent`:** 100%가 아니면 일부 패턴이 인덱싱되지 않은 것입니다(임베딩 누락 혹은 포맷 오류). 이를 모니터링 지표로 삼아 **데이터 파이프라인의 건전성**을 체크합니다.
    

### 5.5 하이브리드 검색 준비 (Hybrid Search Readiness)

벡터 인덱스만으로는 부족한 경우가 있습니다.

- _Case:_ "Python 3.12의 최신 문법 패턴을 보여줘"
    
- 벡터 검색은 "3.12"라는 숫자의 정확한 매칭에 약합니다.
    
- 따라서 Part 3에서 만든 **Fulltext Index(키워드 검색)**와 Part 5의 **Vector Index(의미 검색)**를 결합할 준비를 합니다. 인덱스 설정 시 두 인덱스가 서로 충돌하지 않도록 명확히 분리합니다.

## Part 6: 시맨틱 RAG 검색 알고리즘 (RAG Core)

RAG(Retrieval-Augmented Generation)의 승패는 '생성(Generation)'이 아닌 **'검색(Retrieval)'**에서 갈립니다. 쓰레기를 찾아오면 LLM은 쓰레기를 그럴듯하게 포장할 뿐입니다. 이 시스템의 검색 알고리즘은 **"의미적 유사성(Semantic Similarity)"**과 **"메타데이터 필터링(Metadata Filtering)"**을 결합하여 정확도를 극대화합니다.

### 6.1 검색 워크플로우 아키텍처 (The Retrieval Flow)

사용자가 자연어로 질문을 던지는 순간부터 결과가 반환되기까지의 0.5초를 프레임 단위로 쪼개 분석합니다.

```
User Query ("Python 비동기 에러 잡는 법")
      │
      ▼
[1. Query Pre-processing] ──▶ "Error: Python 비동기 에러..." (Prefix Injection)
      │
      ▼
[2. Query Embedding] ──▶ [0.02, -0.15, ..., 0.88] (1536 dim Vector)
      │
      ▼
[3. Vector Search (Neo4j)] ──▶ HNSW 탐색 (Cosine Similarity)
      │                       └── Filter: Language='python' (Optional)
      ▼
[4. Scoring & Ranking] ──▶ Score > 0.5 인 것만 필터링, 점수순 정렬
      │
      ▼
[5. Result Formatting] ──▶ Markdown으로 변환하여 Claude에게 전달
```

### 6.2 쿼리 처리 전략 (Query Strategy)

사용자의 질문을 그대로 임베딩하지 않습니다. 검색 의도에 따라 **쿼리 성형(Query Shaping)**을 수행하여 벡터 공간에서의 '조준점'을 보정합니다.

#### **6.2.1 전략적 접두사 주입 (Prefix Injection)**

명세서의 `search_by_error`와 `search_by_problem` 메서드를 보면, 쿼리 앞에 특정 키워드를 강제로 붙입니다.

- **에러 검색 시:** `f"Error: {error_message}. Looking for solutions."`
    
    - **의도:** 단순히 에러 메시지만 넣으면, 학습 데이터에 있는 '에러 정의'나 '에러 로그' 자체가 검색될 수 있습니다. "Looking for solutions"라는 문구를 추가함으로써, 벡터의 방향을 **'문제(Problem)'가 아닌 '해결책(Solution)' 쪽으로 미세 조정**합니다.
        
- **문제 해결 검색 시:** `f"Problem: {problem_description}. Need solution pattern."`
    
    - **의도:** 사용자의 모호한 질문("로그인이 안 돼")을 명확한 패턴 요구 사항으로 구체화합니다.
        

### 6.3 Cypher 검색 쿼리 심층 분석 (The Core Logic)

`semantic_search` 메서드 내부의 Cypher 쿼리는 이 시스템의 심장부입니다. 한 줄씩 엔지니어링 의도를 해석합니다.

Cypher

```
MATCH (p:Pattern)
WHERE p.embedding IS NOT NULL
  AND ($language IS NULL OR p.language = $language)  // [A] Pre-filtering
  AND ($category IS NULL OR p.category = $category)
WITH p, vector.similarity.cosine(p.embedding, $embedding) AS score // [B] Similarity Calc
WHERE score >= $min_score  // [C] Threshold Cut-off
RETURN p, score
ORDER BY score DESC
LIMIT $top_k
```

#### **[A] Pre-filtering vs Post-filtering (성능 최적화)**

- **논리:** 벡터 유사도 계산은 비용이 비쌉니다. 따라서 `$language`나 `$category` 같은 명확한 조건이 있다면, **계산하기 전에 먼저 필터링**해야 합니다.
    
- **구현:** `AND` 절을 통해 Python 질문에는 Python 패턴만 대상으로 하도록 범위를 좁힙니다. 이는 HNSW 인덱스 탐색 범위를 획기적으로 줄여 속도와 정확도를 동시에 높입니다.
    

#### **[B] Cosine Similarity (유사도 계산)**

- **수식:** $\text{similarity} = \frac{A \cdot B}{\|A\| \|B\|}$
    
- **해석:** 쿼리 벡터(A)와 패턴 벡터(B) 사이의 각도가 얼마나 좁은지 계산합니다.
    
    - 1.0: 완벽히 동일한 의미.
        
    - 0.0: 전혀 관계없음.
        
    - -1.0: 정반대의 의미.
        

#### **[C] Threshold Cut-off (품질 관리)**

- **`WHERE score >= $min_score` (기본값 0.5)**
    
- **전략적 중요성:** "아무거나 내놓느니 차라리 모른다고 해라."
    
- 유사도가 낮은(예: 0.3) 패턴을 억지로 추천하면 할루시네이션(거짓 정보)의 원인이 됩니다. **최소 품질 임계값(0.5)**을 설정하여, 관련성이 확실한 정보만 LLM에게 전달하는 **품질 방화벽(Quality Firewall)** 역할을 합니다.
    

### 6.4 검색 결과 데이터 구조 및 포맷팅 (Output Engineering)

검색된 결과(`SearchResult`)를 LLM에게 어떻게 먹여주느냐가 최종 응답의 품질을 결정합니다. 명세서는 **Markdown 포맷**을 채택했습니다.

Python



# Result Formatting Logic
def to_markdown(self) -> str:
    return f"""
### {self.name} (Score: {self.score:.2%})
**Problem:** {self.problem}
**Solution:** {self.solution}
**Rule:** `{self.rule}`
```{self.language}
{self.code_example}
````

"""



* **Context Preservation:** 단순 코드가 아니라 **Problem(문제 상황)**과 **Rule(제약 조건)**을 함께 제공합니다.
    * *효과:* Claude Code가 코드를 생성할 때, "아, 이 패턴은 이런 문제일 때 쓰는 거구나"라고 인지하여 문맥에 맞는 코드를 작성하게 유도합니다.
* **Score Exposure:** 유사도 점수(`78%`)를 LLM에게 보여줍니다.
    * *효과:* LLM이 "이 정보는 78% 정도만 확실하구나"라고 인지하여, 불확실한 경우 사용자에게 되묻거나 보수적으로 답변하도록 메타 인지(Meta-cognition)를 돕습니다.

### 6.5 다형성 검색 API (Polymorphic Search Interface)

하나의 검색 로직을 여러 상황에 맞게 오버로딩(Overloading)하여 유연성을 확보했습니다.

1.  **Semantic Search:** 일반적인 "어떻게 해?" 질문 처리.
2.  **Error Search:** "TypeError 발생" 같은 명확한 버그 리포트 처리. 에러 메시지 자체를 쿼리로 사용.
3.  **Related Pattern Search:** "이거랑 비슷한 거 찾아줘."
    * 이 경우, 사용자의 텍스트 쿼리 대신 **현재 보고 있는 패턴의 벡터**를 입력값으로 넣어, 벡터 공간상에서 주변에 있는 패턴들을 찾아냅니다. (Item-to-Item Recommendation)

## Part 7: 에러 로깅 및 해시 시스템 (Error Fingerprinting)

"똑같은 에러를 두 번 고치지 마라." 이것이 이 시스템의 핵심 철학입니다. Part 7은 실패(Error)를 데이터베이스화하여, 개인의 경험을 시스템의 지식으로 승화시키는 메커니즘을 다룹니다.

### 7.1 에러의 자산화 아키텍처 (The Architecture of Failure)

일반적인 로깅 시스템(ELK Stack 등)은 '모니터링'이 목적이지만, 이 시스템은 **'해결(Resolution)'**이 목적입니다.

```
[Runtime Error] 
      │
      ▼
[Capture Context] ──▶ Type, Message, Stack Trace, Local Vars
      │
      ▼
[Fingerprinting] ──▶ Generate Unique ID (MD5 Hash)
      │
      ▼
[Deduplication Check] ──▶ Neo4j 조회: "이 에러 본 적 있어?"
      │                  ├── YES: 발생 횟수(Count) 증가, 타임스탬프 갱신
      │                  └── NO:  새로운 Error Node 생성 및 임베딩
      ▼
[Solution Link] ──▶ (추후 해결 시) Solution 텍스트 업데이트
```

### 7.2 에러 핑거프린팅: MD5 해시 전략 (The Unique ID)

수천 개의 에러 로그 중에서 "이것과 저것은 같은 에러다"라고 기계가 판단하려면, 텍스트 비교가 아닌 **고유 ID**가 필요합니다. 명세서는 이를 위해 **MD5 해시**를 사용합니다.

#### **7.2.1 해시 생성 로직 분석**

Python

```
# 명세서 코드 발췌 및 분석
error_id = hashlib.md5(
    f"{error_type}:{error_message}".encode()
).hexdigest()[:16]
```

- **입력 조합 (`error_type` + `error_message`):**
    
    - `TypeError: cannot unpack non-iterable NoneType object`
        
    - 이 조합이 같으면, 서로 다른 파일에서 발생했더라도 논리적으로 '동일한 유형의 문제'로 간주합니다.
        
    - _전략적 의도:_ 스택 트레이스 전체를 해싱하지 않는 이유는, 줄 번호(Line Number)가 조금만 바뀌어도 다른 에러로 인식되는 것을 막기 위함입니다. 우리는 **'증상(Symptom)'**을 식별하고 싶은 것이지, 특정 코드 라인을 식별하려는 게 아닙니다.
        
- **알고리즘 (MD5):**
    
    - 암호학적 보안이 필요한 게 아니므로, 속도가 빠르고 충돌 확률이 충분히 낮은 MD5를 사용합니다.
        
    - 길이를 16자로 잘라서 인덱싱 성능을 최적화합니다.
        

#### **7.2.2 멱등성(Idempotency) 보장**

이 해시 ID 덕분에 시스템은 **멱등성**을 가집니다. 사용자가 같은 에러를 100번 발생시켜도, DB에는 단 하나의 `Error` 노드만 존재하며 `occurrence_count`만 100으로 증가합니다. 이는 DB 용량 폭발을 막고 학습 데이터의 밀도를 높입니다.

### 7.3 스키마 엔지니어링: Error Node

Part 3의 Pattern 노드와 달리, Error 노드는 **'문제 상황'**을 정밀하게 기록하도록 설계되었습니다.

|**속성**|**설명**|**예시 데이터**|
|---|---|---|
|`error_id`|고유 식별자 (PK)|`a1b2c3d4e5f67890`|
|`error_type`|에러 클래스|`ValueError`|
|`error_message`|에러 메시지 상세|`invalid literal for int() with base 10: 'abc'`|
|`context`|**[핵심]** 발생 상황 설명|"사용자 입력값을 int로 변환하려다 실패함. 입력값이 빈 문자열일 수도 있음."|
|`solution`|해결 방법 (Code)|`try-except 블록으로 감싸고 default 값 0 할당`|
|`tags`|분류 태그|`["casting", "input-validation", "python"]`|
|`embedding`|벡터 데이터|(1536차원 벡터)|

- **`context`의 중요성:** 에러 메시지만으로는 부족합니다. "왜(Why)" 발생했는지를 담는 이 필드가 나중에 RAG 검색 시 결정적인 힌트가 됩니다.
    
- **`solution`의 가변성:** 처음 에러가 등록될 때 이 필드는 비어있을 수 있습니다(`null`). 사용자가 문제를 해결하면(Part 10 자동화), 그때 이 필드가 채워지면서 **완전한 지식(Knowledge)**이 됩니다.
    

### 7.4 에러 벡터화 및 하이브리드 검색

에러 로그도 검색 대상입니다. 사용자가 "숫자 변환이 안 돼"라고 검색하면, 과거에 발생했던 `ValueError` 기록을 찾아줘야 합니다.

#### **7.4.1 에러 임베딩 텍스트 생성**

Python

```
def _prepare_error_text(self, record):
    return f"""
    Error Type: {record.error_type}
    Message: {record.error_message}
    Context: {record.context}
    Solution: {record.solution}  # 해결책이 있다면 포함
    """
```

- 해결책(`solution`)이 포함된 에러 로그는 '패턴'과 동급의 가치를 가집니다. 따라서 임베딩 시 해결책을 포함하여, 유사한 문제를 겪는 사람에게 "이거 이렇게 고쳤었어"라고 알려줍니다.
    

#### **7.4.2 통합 검색 전략 (Logged Error + Pattern)**

명세서의 `search_solutions` 메서드는 두 가지 소스를 동시에 뒤집니다.

1. **Source A: Error Nodes (경험적 지식)**
    
    - "지난주에 김대리가 겪었던 그 에러인가?"
        
    - 실제 프로젝트의 문맥이 반영된 구체적인 해결책.
        
2. **Source B: Pattern Nodes (교과서적 지식)**
    
    - "일반적으로 이런 에러는 어떻게 잡지?"
        
    - 글로벌 베스트 프랙티스.
        

이 두 결과를 합쳐(`extend`) 점수순으로 정렬(`sort`)하여 사용자에게 제공합니다. 이는 **이론과 경험의 조화**를 이룹니다.

### 7.5 추적성 및 디버깅 지원 (Traceability)

에러 노드는 단순히 저장만 되는 게 아니라, 프로젝트 컨텍스트와 연결됩니다.

Cypher

```
MATCH (p:Project {name: "my-app"})
MATCH (e:Error {error_id: "..."})
MERGE (p)-[:ENCOUNTERED {at: datetime()}]->(e)
```

- **프로젝트별 에러 리포트:** "이번 프로젝트에서는 `Async` 관련 에러가 유독 많이(30%) 발생하네?"라는 통찰을 제공할 수 있습니다.
    
- **재발 방지:** 이미 해결된 에러(`solution`이 있는 노드)가 다시 연결되면, 시스템은 즉시 "이거 전에 해결했던 문제입니다. 이 코드를 쓰세요."라고 **0초 만에 답을 제시**할 수 있습니다.

## Part 8: 하이브리드 솔루션 추론 (Hybrid Reasoning)

시스템이 똑똑해 보이려면 단순한 검색 결과 나열이 아니라, **"이 상황에서는 이게 정답입니다"**라고 판단해 주어야 합니다. 하이브리드 추론 엔진은 **귀납적 추론(과거 경험)**과 **연역적 추론(이론적 패턴)**을 결합하여 최적의 솔루션을 도출합니다.

### 8.1 듀얼 브레인 아키텍처 (The Dual-Brain Architecture)

이 시스템은 문제를 해결할 때 두 개의 뇌를 동시에 사용합니다.

```
┌─────────────────────────────────────────────────────────────┐
│                    Hybrid Reasoning Engine                   │
└──────────────────────────────┬──────────────────────────────┘
                               │
               ┌───────────────┴───────────────┐
               ▼                               ▼
      [Left Brain: Experience]        [Right Brain: Theory]
      (Error Node Search)             (Pattern Node Search)
      "우리 팀이 겪었던 문제인가?"     "업계 표준 해결책은 무엇인가?"
      - Source: Local DB              - Source: Global Knowledge
      - Context: High                 - Context: Low
      - Specificity: High             - Specificity: Low
               │                               │
               └───────────────┬───────────────┘
                               ▼
                    [Fusion & Re-ranking]
                    "점수 보정 및 통합 정렬"
                               │
                               ▼
                       [Final Suggestion]
```

### 8.2 병렬 검색 및 병합 전략 (Parallel Retrieval & Merge)

명세서의 `search_solutions` 메서드는 이 병렬 처리를 구현하고 있습니다.

Python

```
def search_solutions(self, error_message, ...):
    suggestions = []
    
    # 1. Experience Retrieval (가중치 높음)
    local_hits = self._search_logged_errors(query_embedding)
    
    # 2. Theory Retrieval (보완재)
    global_hits = self._search_patterns(query_embedding)
    
    # 3. List Merge
    suggestions.extend(local_hits)
    suggestions.extend(global_hits)
```

- **전략적 의도:**
    
    - **Priority to History:** 내 프로젝트에서 발생했던 에러(`local_hits`)는 100% 문맥이 일치할 확률이 높습니다. 따라서 검색 결과 상위에 노출되어야 합니다.
        
    - **Fallback to Theory:** 만약 겪어본 적 없는 새로운 에러라면, `local_hits`는 0건일 것입니다. 이때 시스템은 자연스럽게 `global_hits`(패턴)를 제시하여 "비록 처음 보지만, 보통 이렇게 해결합니다"라고 제안합니다. 이것이 **Cold Start 문제**를 해결합니다.
        

### 8.3 랭킹 알고리즘 및 점수 보정 (The Ranking Logic)

단순히 유사도(Cosine Similarity)만으로 정렬하면 불합리한 경우가 생깁니다. 우리는 **가중치 보정(Re-ranking)** 공식을 적용해야 합니다.

$$FinalScore = (S_{vec} \times W_{type}) + (B_{recency}) + (B_{project})$$

- $S_{vec}$: 벡터 유사도 (0.0 ~ 1.0)
    
- $W_{type}$: 소스 유형 가중치
    
    - **Logged Error:** 1.2 (경험 우대)
        
    - **Global Pattern:** 1.0 (기본)
        
- $B_{recency}$: 최신성 보너스 (Error 노드에만 적용)
    
    - 최근 1주일 내 해결된 에러라면 +0.1점. (오래된 레거시 정보보다 최신 해결책 선호)
        
- $B_{project}$: 프로젝트 일치 보너스
    
    - 현재 작업 중인 프로젝트(`project_id` 일치)에서 발생한 에러라면 +0.2점.
        

**[시나리오 예시]**

- **후보 A (과거 에러):** 벡터 유사도 0.85, 우리 프로젝트, 어제 해결함.
    
    - $Score = 0.85 \times 1.2 + 0.1 + 0.2 = 1.32$
        
- **후보 B (글로벌 패턴):** 벡터 유사도 0.88, Django 공식 패턴.
    
    - $Score = 0.88 \times 1.0 = 0.88$
        
- **결과:** 벡터 유사도는 B가 높지만, 문맥 적합도가 높은 A가 최종 1순위로 추천됩니다.
    

### 8.4 충돌 해결 및 다양성 확보 (Conflict Resolution)

만약 Top 5 결과가 모두 똑같은 해결책(중복)이라면 정보의 가치가 떨어집니다. 반대로, 상반된 해결책이 나올 수도 있습니다.

#### **8.4.1 MMR (Maximal Marginal Relevance) 적용**

결과 리스트의 다양성을 확보하기 위해 MMR 알고리즘의 개념을 차용합니다.

1. 가장 점수가 높은 1위를 선정합니다.
    
2. 2위부터는 **1위와 너무 비슷하면(유사도 > 0.95) 탈락**시킵니다.
    
3. **이유:** "서로 다른 관점의 해결책"을 제공하기 위함입니다.
    
    - _Solution 1:_ "라이브러리 버전을 올리세요."
        
    - _Solution 2:_ "코드의 비동기 로직을 수정하세요."
        
    - 이 두 가지를 모두 보여줘야 사용자가 선택할 수 있습니다.
        

### 8.5 최종 응답 구조화 (The Output Interface)

추론 엔진의 결과물은 LLM(Claude)이 이해하고 사용자에게 설명하기 좋은 구조여야 합니다.

Python

```
@dataclass
class SolutionSuggestion:
    source_type: str  # "HISTORY" | "PATTERN"
    title: str
    content: str
    confidence_score: float
    metadata: dict    # { "project": "my-app", "solved_at": "2025-..." }

    def format_for_llm(self):
        if self.source_type == "HISTORY":
            return f"[✅ Solved Record] (Confidence: {self.confidence_score:.2f})\n" \
                   f"This error occurred in {self.metadata['project']} and was fixed by:\n" \
                   f"{self.content}"
        else:
            return f"[💡 Design Pattern] (Confidence: {self.confidence_score:.2f})\n" \
                   f"Recommended architectural approach:\n" \
                   f"{self.content}"
```

- **LLM 페르소나 주입:**
    
    - `HISTORY`일 경우: "동료 개발자가 옆에서 알려주는 톤"으로 설명하도록 유도.
        
    - `PATTERN`일 경우: "시니어 아키텍트가 조언하는 톤"으로 설명하도록 유도.

## Part 9: Claude Code Skills 통합 구조 (Integration Interface)

Claude Code의 **Skills** 기능은 LLM에게 "너는 이제부터 이 도구를 쓸 수 있어"라고 알려주는 API 명세서와 같습니다. 핵심은 Claude가 **언제(When)**, **어떻게(How)** 이 도구를 써야 하는지 자연어로 이해시키는 것입니다.

### 9.1 Skills 디렉토리 및 파일 아키텍처

Claude Code는 로컬 파일시스템의 특정 경로를 읽어 자신의 능력을 확장합니다.

```
~/.claude/skills/
└── knowledge-graph-rag/          # 스킬 패키지 (Namespace)
    ├── SKILL.md                  # [핵심] 스킬 정의서 (AI가 읽는 매뉴얼)
    ├── rag_client.py             # RAG 검색 실행기 (Part 6)
    ├── error_client.py           # 에러 로깅/검색 실행기 (Part 7)
    ├── requirements.txt          # 의존성 목록
    └── .env                      # API 키 및 DB 접속 정보
```

- **격리(Isolation):** 각 스킬은 폴더별로 격리되어 관리됩니다.
    
- **`SKILL.md`의 역할:** 이 파일은 사람을 위한 문서가 아니라, **Claude를 위한 시스템 프롬프트**입니다. 여기에 적힌 내용이 Claude의 Context Window에 주입되어 행동을 제어합니다.
    

### 9.2 SKILL.md 심층 분석: AI 제어 프로토콜

`SKILL.md`는 단순한 설명서가 아닙니다. **입력(Intent)을 명령어(Command)로 변환하는 프로토콜**입니다.

#### **9.2.1 구조적 정의 (The Anatomy of SKILL.md)**

Markdown

```
# Knowledge Graph RAG (Tool Definition)

## Description
This skill allows you to retrieve architectural patterns and solutions for runtime errors from a Neo4j Knowledge Graph containing 12,960+ verified patterns.
(이 설명은 Claude가 "내가 이 도구를 써야겠구나"라고 판단하는 근거가 됩니다.)

## Commands (The Interface)

### 1. search_pattern
- **Usage:** `python3 rag_client.py --query "{query}" --top 5`
- **Description:** Use this when the user asks about design patterns, best practices, or "how to implement X".
- **Parameter Rules:**
  - `query`: Must be in English for better embedding matching. Translate if user asks in Korean.
  # [전략적 포인트] 한국어 질문이 들어와도 검색 정확도를 위해 영어로 변환하여 쿼리하라는 지시를 포함.

### 2. resolve_error
- **Usage:** `python3 error_client.py search --error "{error_msg}" --language "{lang}"`
- **Description:** TRIGGER IMMEDIATELY when a traceback or error message is detected.
- **Trigger Keywords:** `Error`, `Exception`, `Failed`, `Crash`

## Output Format (The Protocol)
The tool returns JSON. You MUST parse this JSON and present it to the user in the following Markdown format...
(Python 스크립트가 뱉은 JSON을 사람이 읽기 좋은 형태로 가공하는 방법을 지시합니다.)
```

### 9.3 프롬프트 엔지니어링 전략 (Prompt Engineering for Agents)

AI가 도구를 멍청하게 쓰지 않도록 제어하는 **가드레일(Guardrails)**을 `SKILL.md`에 심어야 합니다.

#### **9.3.1 자동 번역 및 쿼리 최적화**

- **문제:** 사용자가 "비동기 처리 어떻게 해?"라고 물음.
    
- **임베딩 데이터:** 대부분 영어(`Async handling...`).
    
- **해결:** `SKILL.md`에 **"Translate non-English queries to English before execution"** 규칙을 명시하여 검색 품질을 강제로 높입니다.
    

#### **9.3.2 실행 판단 로직 (Decision Logic)**

Claude가 언제 검색을 수행할지 기준을 정해줍니다.

- **Bad:** 사용자가 "안녕" 했는데 검색 스크립트 실행 -> 자원 낭비.
    
- **Good:** "관련된 패턴 찾아줘"라는 명시적 요청이나, 대화 중 코드가 실패했을 때만 실행.
    
- **구현:** `SKILL.md` 내 `## Auto-Trigger Conditions` 섹션에 정규식 패턴(`Error:.*`, `Traceback.*`)을 명시하여 조건부 실행을 유도합니다.
    

### 9.4 I/O 데이터 파이프라인 (The Data Bridge)

Claude(Text)와 Python Script(Code) 사이의 데이터 변환 과정을 분석합니다.

```
[Claude Code]
   "User wants to fix a TypeError."
        │
        ▼ (Translation based on SKILL.md)
   [Shell Command Execution]
   $ python3 error_client.py search --error "TypeError: ..."
        │
        ▼ (Python Script Logic)
   [JSON Output to stdout]
   {
     "status": "success",
     "hits": [
       {"title": "Check NoneType", "score": 0.92, ...}
     ]
   }
        │
        ▼ (Parsing)
[Claude Code]
   "JSON을 읽어보니 해결책이 있군요. 
    가장 높은 점수(92%)를 받은 'NoneType 체크' 방법을 제안합니다..."
```

- **JSON의 중요성:** 스크립트가 사람에게 보여줄 텍스트(Markdown)를 직접 출력하면, Claude가 내용을 재가공하기 어렵습니다. **구조화된 데이터(JSON)**를 뱉게 하여 Claude가 상황에 맞게("간략히 말해줘" vs "자세히 말해줘") 답변을 구성할 자유를 줍니다.
    

### 9.5 보안 및 권한 제어 (Security Context)

Skills는 내 컴퓨터에서 코드를 실행하는 권한을 가집니다. 따라서 최소 권한 원칙을 적용해야 합니다.

- **Read-Only:** `rag_client.py`는 DB 읽기 권한만 가짐.
    
- **Write-Controlled:** `error_client.py`의 `log` 기능은, 사용자가 명시적으로 "이거 기억해둬"라고 하거나 자동화 설정이 켜져 있을 때만(Part 10) 실행되도록 플래그 처리.
    
- **Sandbox:** 모든 스크립트는 가상 환경(`venv`) 내에서 실행되어 시스템 전역 패키지를 오염시키지 않음.

## Part 10: 프로젝트 컨텍스트 인식 및 훅 시스템 (Context Awareness & Hook System)

진정한 '바이브코딩(Vibe Coding)'은 개발자가 말하지 않아도 AI가 이미 상황을 알고 있을 때 가능합니다. 이를 위해 우리는 **상태 관리(State Management)**와 **이벤트 후킹(Event Hooking)**이라는 두 가지 핵심 기술을 도입합니다.

### 10.1 전체 자동화 아키텍처 (The Nervous System)

이 시스템은 개발 환경에서 발생하는 모든 신호(Signal)를 감청합니다.

```
┌────────────────────┐      ┌─────────────────────────────┐      ┌────────────────────┐
│   IDE / Terminal   │      │      Claude Code Core       │      │   Skill Engine     │
│  (The Environment) │      │      (The Processor)        │      │   (The Actor)      │
└─────────┬──────────┘      └──────────────┬──────────────┘      └──────────┬─────────┘
          │                                │                                │
          │ 1. User Action (Save/Run)      │                                │
          ├───────────────────────────────▶│                                │
          │                                │ 2. Hook Trigger (Match?)       │
          │                                │─── [Regex / Exit Code] ───▶    │
          │                                │                                │ 3. Auto-Execute
          │                                │                                │─── [Search RAG]
          │                                │                                │─── [Update Context]
          │                                │◀───────────────────────────────┘
          │ 4. Proactive Suggestion        │
          │◀───────────────────────────────┘
```

### 10.2 프로젝트 컨텍스트 인식 (Context Awareness)

LLM은 기억력이 없습니다(Stateless). 따라서 우리는 **`project_context.json`**이라는 외부 기억장치를 통해 프로젝트의 '상태'를 유지합니다.

#### **10.2.1 컨텍스트 파일 구조 분석**

명세서의 `project_context.json`은 단순한 로그가 아니라 **프로젝트의 생체 신호(Vitals)**입니다.

JSON

```
{
  "project_id": "uuid-v4",

#### **10.2.2 컨텍스트 감지기 (The Detector)**

`detector.py` 스크립트는 프로젝트 루트의 설정 파일(`package.json`, `pyproject.toml`, `Cargo.toml`)을 파싱하여 기술 스택을 자동으로 식별합니다. 이 과정은 Claude Code가 실행될 때 **최초 1회** 자동으로 수행됩니다.

### 10.3 훅 시스템 설계 (Hook System Mechanics)

훅(Hook)은 특정 이벤트 발생 시 자동으로 스킬을 실행하게 만드는 **트리거(Trigger)**입니다. `settings.json`에 정의된 규칙들이 어떻게 작동하는지 분석합니다.

#### **10.3.1 UserPromptSubmit 훅 (의도 감지)**

사용자가 엔터를 치는 순간(Prompt Submit), LLM에게 가기 전에 훅이 먼저 가로챕니다.

JSON

```
"UserPromptSubmit": [
  {
    "matcher": "(?i)(error|exception|failed|안돼|에러|오류)",
    "action": "skill:knowledge-graph-rag/search_error"
  }
]
```

- **작동 원리:** 정규식(Regex)으로 키워드를 감시합니다.
    
- **효과:** 사용자가 "아 이거 왜 에러나지"라고 입력하면, Claude가 답변을 생성하기 **전에** 이미 RAG 검색이 백그라운드에서 실행됩니다. Claude는 이미 검색된 해결책을 프롬프트 컨텍스트로 받은 상태에서 첫 마디를 시작할 수 있습니다.
    
    - _Before:_ "에러 내용을 알려주시겠어요?" (1턴 낭비)
        
    - _After:_ "말씀하신 에러는 Next.js 설정 문제로 보입니다. 관련된 해결책을 가져왔습니다." (Zero-latency)
        

#### **10.3.2 PostToolUse 훅 (결과 감지)**

Claude가 터미널 명령어(Bash)를 실행한 **직후**의 결과를 감시합니다. 이것이 진정한 자동화의 핵심입니다.

JSON

```
"PostToolUse": [
  {
    "tool": "Bash",
    "exitCode": "!0",  // [핵심] Exit Code가 0이 아님 (즉, 실패함)
    "action": "skill:knowledge-graph-rag/analyze_error"
  }
]
```

- **시나리오:**
    
    1. Claude가 테스트 코드 실행 (`npm test`).
        
    2. 테스트 실패 (`Exit Code: 1`).
        
    3. **자동 트리거:** 훅이 이를 감지하고 즉시 `analyze_error` 스킬 실행.
        
    4. 스킬은 실패 로그(`stderr`)를 캡처하여 RAG 검색 수행.
        
    5. **결과:** 사용자가 "실패했네, 분석해줘"라고 말할 필요 없이, 시스템이 "테스트가 실패했습니다. 로그를 분석해보니 DB 연결 타임아웃이네요. 해결 패턴을 찾았습니다."라고 선제 보고.
        

### 10.4 Shadow Mode와 안전장치 (Safety)

자동화는 편리하지만 위험할 수 있습니다. 훅 시스템은 **Shadow Mode** 설정을 지원합니다.

- **Active Mode:** 훅이 트리거되면 즉시 액션 실행 및 결과 출력.
    
- **Shadow Mode:** "이 훅을 실행할까요?"라고 사용자에게 승인 요청 (Human-in-the-loop).
    
- **전략:** 읽기 전용 작업(RAG 검색)은 Active Mode로, 쓰기 작업(코드 수정, 에러 학습)은 Shadow Mode로 설정하여 안전성을 확보합니다.

## Part 11: 자기 주도 학습 시스템 (Auto-Learning Flywheel)

이 시스템의 가장 강력한 경쟁력은 **"오늘의 실패가 내일의 지식이 된다"**는 점입니다. 이를 위해 우리는 **AutoLearner**라는 백그라운드 프로세스를 가동합니다. 이것은 단순한 로거가 아니라, 인과관계(Causality)를 추론하는 분석 엔진입니다.

### 11.1 학습 파이프라인 아키텍처 (The Learning Loop)

학습 과정은 4단계의 상태 기계(State Machine)로 작동합니다.

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  1. Detection   │     │  2. Tracking    │     │  3. Validation  │     │  4. Assetization│
│  (에러 감지)      │───▶ │  (수정 추적)     │───▶ │  (해결 검증)     │───▶ │  (지식 자산화)   │
│  "TypeError!"   │     │  "코드 변경됨"    │     │  "Test Pass"    │     │  "Neo4j 저장"   │
└─────────────────┘     └─────────────────┘     └─────────────────┘     └─────────────────┘
         ▲                                                │
         └────────────────── (Feedback) ──────────────────┘
```

### 11.2 상태 추적 로직: `AutoLearner` Class

명세서에 정의된 `AutoLearner` 클래스는 이 복잡한 상태를 메모리에서 관리합니다. 핵심 메서드들의 작동 원리를 분석합니다.

#### **11.2.1 에러 인스턴스 생성 (`on_error_detected`)**

- **Trigger:** Part 10의 훅 시스템이 에러를 감지했을 때.
    
- **Logic:**
    
    - 에러의 해시 ID를 생성합니다.
        
    - `pending_errors` 딕셔너리에 에러 상태를 'Open'으로 기록합니다.
        
    - **Snapshot:** 에러 발생 시점의 코드 스냅샷을 저장합니다. (나중에 "Before & After" 비교를 위해)
        

#### **11.2.2 해결 시도 추적 (`on_solution_attempt`)**

- **Trigger:** 사용자가 파일을 수정하고 저장(`PostToolUse: Write`)했을 때.
    
- **Logic:**
    
    - 에러가 발생한 파일이 수정되었는가? (Yes -> 연관성 높음)
        
    - 수정된 코드를 `attempts` 리스트에 타임스탬프와 함께 기록합니다.
        
    - 이는 나중에 **"어떤 수정이 결정적이었는가?"**를 판단하는 근거가 됩니다.
        

### 11.3 검증 알고리즘: "해결됨"을 어떻게 아는가? (Validation Logic)

기계가 "에러가 고쳐졌다"고 판단하는 것은 매우 어렵습니다. 이 시스템은 **3가지 휴리스틱(Heuristics)**을 결합하여 판단합니다.

1. **Explicit Verification (테스트 통과):**
    
    - 사용자가 수정한 뒤 `npm test`나 `python test.py`를 실행하고, **Exit Code 0**이 나오면 즉시 "Resolved"로 판정합니다. (가장 신뢰도 높음)
        
2. **Implicit Stability (시간적 안정성):**
    
    - 코드를 수정한 뒤 **10분 동안 동일한 에러 로그가 발생하지 않음** + **사용자가 다른 파일로 작업을 이동함**.
        
    - -> "아마도 해결되었으니 넘어갔을 것이다"라고 추론합니다.
        
3. **Semantic Check (의미적 확인):**
    
    - 에러가 났던 라인이 삭제되거나 완전히 재작성되었는가?
        

### 11.4 지식 자산화 (Assetization)

검증이 완료되면, 메모리에 있던 데이터를 영구적인 지식(Neo4j)으로 승격시킵니다.

#### **11.4.1 데이터 정제 및 저장**

Python

```
def on_error_resolved(self, error_hash: str):
    # 1. 결정적 해결책(Last Attempt) 추출
    last_attempt = error_info["attempts"][-1]
    
    # 2. 노이즈 제거 (diff 분석)
    # 전체 파일이 아니라, 에러 라인 주변의 변경사항(Diff)만 추출하여 솔루션으로 저장
    solution_diff = self._generate_focused_diff(
        before=error_info["snapshot"], 
        after=last_attempt["code_change"]
    )
    
    # 3. Neo4j 저장 (Auto-learned 태그 부착)
    self.error_logger.log_error(
        ...,
        solution=solution_diff,
        tags=["auto-learned", "verified"]
    )
```

- **전략적 중요성:** `tags=["auto-learned"]`를 붙이는 이유는 신뢰도 등급 때문입니다.
    
    - 사람이 검수한 패턴(신뢰도 A) vs AI가 자동 학습한 패턴(신뢰도 B).
        
    - 나중에 검색 시 가중치를 다르게 적용하거나, **사람이 주기적으로 검수(Review)**할 때 필터링하기 위함입니다.
        

### 11.5 플라이휠 효과 (The Flywheel Effect)

이 시스템은 사용할수록 기하급수적으로 강력해집니다.

- **Day 1:** 지식이 패턴 12,960개뿐임.
    
- **Day 30:** 개발자 A가 겪은 `Next.js 14` 최신 버그 해결책이 자동 학습됨.
    
- **Day 31:** 개발자 B가 같은 버그를 겪자마자 시스템이 **"어제 A님이 이렇게 해결했습니다"**라고 즉시 답을 줌.
    
- **Result:** 팀 전체의 삽질 시간이 0으로 수렴하게 됩니다. 이것이 **조직적 지능(Organizational Intelligence)**의 완성입니다.

## Part 12: 보안, 안정성 및 관측 가능성 (SecOps & Observability)

우리는 지금까지 강력한 엔진(AI Logic)을 만들었습니다. Part 12는 이 엔진이 과열되지 않고 안전하게 주행하도록 돕는 **브레이크, 에어백, 그리고 계기판**을 만드는 과정입니다.

### 12.1 보안 아키텍처: 방어 심층 (Defense in Depth)

명세서는 단순한 API Key 관리를 넘어, 3단계 방어 체계를 구축했습니다.

#### **12.1.1 시크릿 관리 (Secret Management)**

Python

```
class SecretManager:
    def __init__(self):
        # 환경변수조차 평문으로 저장하지 않음. 마스터 키로 한 번 더 암호화.
        self.cipher = Fernet(os.environ["MASTER_KEY"].encode())

    def get_secret(self, key: str) -> str:
        encrypted = os.environ.get(f"{key}_ENCRYPTED")
        return self.cipher.decrypt(encrypted).decode()
```

- **Why?** `.env` 파일이 깃허브에 실수로 올라가는 사고는 빈번합니다. 환경변수 값을 암호화된 문자열(`gAAAAABk...`)로 저장해두면, 파일이 유출되어도 `MASTER_KEY` 없이는 복호화가 불가능하여 **API Key 탈취를 원천 봉쇄**합니다.
    

#### **12.1.2 Cypher Injection 방지**

SQL Injection처럼 그래프 DB도 **Cypher Injection** 공격에 취약합니다.

- **취약한 코드:** `query = f"MATCH (n) WHERE n.name = '{user_input}'"` (위험)
    
- **방어 코드:** `session.run("MATCH (n) WHERE n.name = $name", name=user_input)`
    
- **전략:** 모든 쿼리는 반드시 파라미터 바인딩(`$variable`)을 사용하도록 `safe_query` 래퍼 함수를 강제합니다.
    

#### **12.1.3 프로젝트 격리 (Multi-tenancy Isolation)**

여러 프로젝트를 동시에 관리할 때, A 프로젝트의 코드가 B 프로젝트 검색 결과에 나오면 **심각한 정보 유출**입니다.

- **논리적 격리:** 모든 검색 쿼리에 `MATCH (proj:Project {id: $current_id})-[:OWNS]->(pattern)` 조건을 강제 삽입하여, 해당 프로젝트 소유의 데이터만 접근하도록 제한합니다.
    

### 12.2 안정성 엔지니어링: 서킷 브레이커 (Circuit Breaker)

외부 의존성(OpenAI API, Neo4j Aura)은 언제든 실패할 수 있습니다. 시스템 전체가 멈추지 않도록 **차단기(Circuit Breaker)** 패턴을 구현합니다.

#### **12.2.1 상태 기계 (State Machine) 분석**

- **CLOSED (정상):** API 호출이 정상적으로 통과됩니다.
    
- **OPEN (차단):** 최근 5번 연속 실패 시 회로가 열립니다.
    
    - 이때부터는 API를 호출하지 않고 **즉시 에러를 반환하거나 캐시된 데이터를 반환(Fallback)**합니다. 불필요한 대기 시간(Timeout)을 없애 사용자 경험을 보호합니다.
        
- **HALF-OPEN (정찰):** 일정 시간(300초) 후, 딱 한 번 API를 호출해봅니다. 성공하면 CLOSED로 복구, 실패하면 다시 OPEN으로 돌아갑니다.
    

#### **12.2.2 우아한 성능 저하 (Graceful Degradation)**

서킷이 열렸을 때 시스템이 죽는 게 아니라, **"덜 똑똑하지만 작동은 하는"** 상태로 전환됩니다.

- **Plan A (Full RAG):** OpenAI 임베딩 + Neo4j 벡터 검색 (최고 성능).
    
- **Plan B (Local Fallback):** 네트워크가 끊기면 로컬 SQLite 캐시나 간단한 키워드 검색으로 전환하여 **최소한의 응답**을 보장합니다.
    

### 12.3 비용 공학 (FinOps & Cost Control)

LLM API 비용은 순식간에 불어날 수 있습니다. `CostController`는 이를 통제하는 CFO(재무책임자) 역할을 합니다.

#### **12.3.1 임베딩 캐시 (The 70% Savings)**

Python

```
def get_embedding_with_cache(self, text: str) -> list:
    # 1. 해시 키 생성
    text_hash = hashlib.md5(text.encode()).hexdigest()
    
    # 2. 로컬 SQLite 조회
    if cached := self.cache.get(text_hash):
        return cached  # 비용 $0, 시간 1ms
        
    # 3. API 호출 (비용 발생)
    embedding = self.openai.create(...)
    self.cache.set(text_hash, embedding)
    return embedding
```

- **효과:** 코드 패턴이나 에러 메시지는 반복되는 경향이 있습니다. 한 번 검색된 내용은 로컬에 저장하여, 재검색 시 **비용 0원, 지연 시간 1ms**로 처리합니다. 명세서는 이를 통해 **비용 70% 절감**을 예상합니다.
    

#### **12.3.2 일일 예산 한도 (Daily Hard Limit)**

- 개발자의 실수를 방지하기 위해 일일 한도($1.00)를 설정합니다. 한도가 초과되면 `BudgetExceededError`를 발생시켜 추가 과금을 물리적으로 차단합니다.
    

### 12.4 관측 가능성: 대시보드 (Observability)

"측정할 수 없으면 개선할 수 없다." 시스템의 건강 상태를 한눈에 볼 수 있는 대시보드 지표를 설계합니다.

#### **12.4.1 핵심 성과 지표 (KPIs)**

|**지표 (Metric)**|**목표치 (SLO)**|**설명 및 대응**|
|---|---|---|
|**p95 Latency**|**< 450ms**|100명 중 95명의 사용자가 0.45초 이내에 결과를 받아야 함. 초과 시 인덱스 튜닝 필요.|
|**Cache Hit Rate**|**> 70%**|캐시 적중률이 떨어지면 캐시 TTL(수명)을 늘리거나 전략 수정 필요.|
|**Auto-Learn Success**|**> 80%**|자동 학습된 패턴이 검증을 통과하는 비율. 낮으면 검증 로직 강화 필요.|
|**Error Rate**|**< 1%**|시스템 내부 오류 비율. 서킷 브레이커 작동 빈도 모니터링.|

#### **12.4.2 대시보드 UI 구조**

```
┌─────────────────────────────────────────────────────────────┐
│  [Status: ONLINE]  [Budget: $0.23 / $1.00]  [Cache: 78%]    │
├──────────────────────┬──────────────────────┬───────────────┤
│  Latency (ms)        │  RAG Quality         │  Learned      │
│  Current: 120ms      │  👍 Helpful: 142     │  Pattern: +15 │
│  [Graph: Line Chart] │  👎 Bad: 12          │  Error: +3    │
