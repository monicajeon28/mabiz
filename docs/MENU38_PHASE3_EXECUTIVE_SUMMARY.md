### Q2: 데이터 검증 방식?

**추천**: 자동화 (매일 06:00 자동 검사)

| 방식 | 비용 | 신뢰 | 선택 |
|------|------|------|------|
| 자동화 | 💰 2-3시간 | ✅ 99.9% | **✅ 예** |
| 수동 | 💰 저 | ❌ 50% | ❌ 아니오 |
| 대시보드만 | 💰 저 | 🟡 70% | ❌ 아니오 |

**자동 검증 내용** (매일 06:00):
```
✓ 행 수 비교 (SendingHistory vs ExecutionLog)

✓ 실패율 비교
✓ NULL 값 분포
✓ API 응답 시간

결과:
├─ OK → 일반 리포트
├─ 주의 → Slack 알림
└─ 실패 → 긴급 알림 + 롤백 검토
```

**기대 효과**:
- 🔔 자동 알림 (문제 발생 시 < 5분)
- 📊 데이터 기반 의사결정
- 👤 수동 개입 최소화

```
    ↓
```

- ⏱️ < 1분 내 복구

## 🎓 기술 이해 (경영진용)

### 현재 상황
```
SendingHistory (레거시):
├─ 2025-01 ~ 현재: 100만 건 기존 데이터
└─ 계속 새 데이터 기록 중

ExecutionLog (신규):
├─ 더 나은 구조 (자동화/퍼널 지원)
└─ Phase 3부터 새 데이터 기록
```

### 마이그레이션 전략
```
Phase 2 (현재):
├─ SendingHistory만 사용
└─ 문제 없음

Phase 3a (1주):
├─ SendingHistory + ExecutionLog 둘 다 사용 (검증)
├─ 새 캠페인은 ExecutionLog 기록
├─ API 조회: UNION (둘 다 합쳐서)
└─ 검증 통과하면 다음

Phase 3b (1주):
├─ ExecutionLog만 사용 (완전 전환)
├─ 성능 10-20% 개선
└─ 마이그레이션 완료

Phase 3c (정리):
├─ SendingHistory 아카이빙
└─ 인덱스 최적화
```

### 왜 이렇게 할까?

**즉시 전환이 아니라 1주일 더 걸리는 이유:**
- 100만 건 기존 데이터와 신규 데이터 검증 필요
- 버그 발견 시 롤백 가능하게 하기
- 신뢰도 99.9% 달성하기 위함
- **비용**: 추가 1주일
- **효과**: 데이터 손실 0건, 신뢰도 99.9%

**자동 롤백의 중요성:**
- 아무리 준비해도 문제는 발생 가능
- 즉시 복구 능력이 신뢰 구축
- < 1분 내 자동으로 원래 상태로 복구
- 추가 개발 비용 1-2시간
