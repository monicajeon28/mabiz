# 01. 데이터 모델 & 아이디 발급 규칙

> CRM이 어필리에이트를 발급할 때 **이 스펙을 정확히 따라야** 몰이 그대로 이어받아 동작한다.
> 두 시스템은 **같은 Neon DB**를 쓰므로, CRM이 아래 테이블에 row를 정확히 쓰면 몰이 즉시 인식한다.

---

## 1. 발급 시 만들어야 하는 테이블 2개

어필리에이트 "아이디 발급" = **`User` 1행 + `AffiliateProfile` 1행** (1:1, `AffiliateProfile.userId @unique`).

### A. User (로그인 주체)
| 컬럼 | 값/규칙 | 필수 |
|---|---|---|
| `phone` | 발급 아이디(=mallUserId와 동일값). 예: `boss1` | ✅ (로그인 ID) |
| `mallUserId` | `phone`과 동일값. 예: `boss1` | ✅ |
| `password` | **bcrypt 해시** (초기값 평문 `1101`을 해시). ⚠️ 평문 저장 금지 | ✅ |
| `name` / `mallNickname` | 어필리에이트 이름/표시명 | 권장 |
| `role` | `'community'` (어필리에이트 기본) | ✅ |
| `customerSource` | 예: `'crm-contract'` (발급 출처 식별용) | 권장 |
| `customerStatus` | `'pending'` 또는 운영정책값 | 선택 |

> 로그인은 `mallUserId` 또는 `phone`으로 조회 → 둘을 같은 값으로 맞추는 게 안전.
> 비밀번호는 최초 `1101`(또는 합의된 초기값)을 bcrypt 해시로 저장. 변경 이력은 `PasswordEvent`에 기록 권장.

### B. AffiliateProfile (권한·계약·정산 정보)
필수/핵심 필드:
| 컬럼 | 값/규칙 | 비고 |
|---|---|---|
| `userId` | 위 User.id (1:1, unique) | ✅ |
| `type` | `'BRANCH_MANAGER'`(대리점장) / `'SALES_AGENT'`(판매원) / `'PRE_SALES'`(프리세일즈) / `'HQ'`(본사) | ✅ String |
| `status` | 발급완료=`'ACTIVE'` (몰 로그인 통과 조건). 미완료=`'DRAFT'` | ✅ |
| `contractStatus` | 계약완료=`'SIGNED'`, 진행중=`'DRAFT'` | ✅ |
| `affiliateCode` | **고유값**(unique). 추적 링크/귀속 키. 영숫자+`_-` | ✅ |
| `withholdingRate` | 원천징수율(%). 기본 `3.3` | ✅ |
| `agentCommissionRate` | 판매원 기본 수당율(%) (없으면 상품/역할 단가→3% 폴백) | 선택 |
| `displayName` / `nickname` / `branchLabel` | 표시명/닉네임/지점명 | 선택 |
| `contactPhone` / `contactEmail` | 연락처 | 선택 |
| `bankName` / `bankAccount` / `bankAccountHolder` | 정산 계좌 | 정산 시 필요 |
| `guarantorName` / `guarantorId` | 보증인 | 선택 |
| `contractSignedAt` / `contractSignature` / `contractIp` / `contractVersion` / `contractUserAgent` | 전자계약 싸인 증적 | 계약 시 |
| `kycCompletedAt` / `onboardedAt` / `welcomeEmailSentAt` | 온보딩 단계 | 선택 |
| `landingSlug` | 파트너 랜딩 슬러그(고유). 미지정 시 아이디값 사용 | 선택 |
| `metadata` (Json) | CRM 출처 메타 권장: `{ source:'CRM', contractRef, crmLinkCode }` | 권장 |
| `published` | 기본 `true` | |

### C. AffiliateRelation (대리점장↔판매원 관계) — 판매원/프리세일즈 발급 시
| 컬럼 | 값 |
|---|---|
| `managerId` | 소속 대리점장의 **AffiliateProfile.id** |
| `agentId` | 본인(판매원)의 **AffiliateProfile.id** |
| `status` | `'ACTIVE'` |
| (unique) | `(managerId, agentId)` |

> 판매원의 수당/고객 귀속은 이 관계의 `managerId`로 대리점장에게 연결된다. **관계 누락 시 수당 정산이 깨진다.**

## 2. 아이디(mallUserId) 발급 규칙 — 충돌 방지 필수

현재 몰 로직(`generatePartnerId`)과 **동일하게** 구현해야 충돌이 안 난다.

**prefix (역할별):**
- 대리점장(BRANCH_MANAGER) → `boss`
- 판매원(SALES_AGENT) → `user`
- 프리세일즈(소속 없는 판매원) → `free`
- (참고) 몰 계약승인 경로는 `pre`(프리세일즈)/`sales`(어드민직접)도 사용 → **CRM은 prefix 규칙을 하나로 통일하고 그 규칙을 몰과 공유할 것**

**번호 채번(충돌검사):**
1. `User.phone`에서 `prefix`로 시작하는 모든 행 조회 (`startsWith`)
2. 정규식 `^{prefix}(\d{1,5})(?:-.*)?$` 로 숫자 추출 → 사용중 번호 집합
3. **1부터 올라가며 빈 번호** 채택 → `prefix + 번호` (예: `boss1`, `user7`)
4. `mallUserId = phone = prefix+번호`

> ⚠️ 같은 DB라 CRM과 몰이 동시에 채번하면 race 가능 → **발급 주체를 CRM 단독**으로 정하면 해소(옵션 D). CRM이 채번 시 위 쿼리를 그대로 사용.

**affiliateCode:** 이름+userId 기반 고유 코드 생성(몰 `generateAffiliateCode`). CRM도 고유성만 보장하면 자유.

---

## 3. 비밀번호 정책
- 초기 비번 = `1101` (또는 합의값) → **반드시 bcrypt 해시로 저장**.
- 몰의 공유 기본비번(`1101`/`qwe1`/`zxc1`)은 첫 로그인 시 해시 승격되는 레거시 편의값 → CRM 신규 발급은 처음부터 bcrypt 권장.
- 비번 변경/초기화 시 `PasswordEvent`에 기록(평문 미저장).

---

## 4. 발급 후 "부수효과" (몰이 처리해야 정상 동작)
CRM이 User/AffiliateProfile/Relation row만 만들면 **로그인은 되지만** 아래가 비어 파트너몰이 반쪽이 된다 → `03_몰연동규약_provision.md`의 provision API로 몰이 생성:
- 기본 어필리에이트(공유) 링크
- 기본 고객그룹("…의 고객")

## 부록: 다운스트림(이 데이터가 쓰이는 곳)
- **로그인**: `mallUserId/phone + password` → 세션 → `AffiliateProfile(status=ACTIVE)` 인식
- **수당정산**: `withholdingRate`, `agentCommissionRate`, `AffiliateRelation.managerId`
- **문의/고객 귀속**: `AffiliateProfile.id` (대리점장/판매원), 크루즈닷봇 문의도 이 id로 귀속
