# 어필리에이트 인력관리 → CRM 이관 인계 패키지

> 작성 2026-06-07 (크루즈닷몰 → CRM팀 전달용)
> 목적: **계약서 싸인 + 아이디 발급을 CRM에서 담당**하기로 정책 변경됨.
> 현재 몰(`/admin/affiliate/profiles`)의 인력관리가 더 상세하므로, **CRM이 동등 수준으로 먼저 구축·세팅한 뒤** 몰 쪽 이어받기 작업을 진행한다.

## 왜 CRM부터인가
- 몰과 CRM은 **동일한 Neon DB를 공유**한다(SSoT=Neon). → CRM이 `User`/`AffiliateProfile` row를 만들면 몰이 **동기화 없이 즉시 읽는다**(이어받음).
- 단, 현재 코드상 **발급(아이디/비번 생성) 책임이 몰에 4경로로 산재**한다. CRM이 발급 기능을 동등하게 갖추기 전에 몰의 발급을 막으면 "발급할 곳이 없어진다".
- 따라서 순서: **① CRM 발급기능 구축(이 문서들 기준) → ② CRM에서 테스트 발급 → ③ 몰 발급경로 봉인 + provision 연동.**

## 문서 구성
| 파일 | 내용 |
|---|---|
| `01_데이터모델_및_아이디발급.md` | AffiliateProfile/User 필드 전체 + 아이디 발급 규칙(prefix/충돌검사) + 비밀번호 정책 |
| `02_인력관리_기능명세.md` | 몰 인력관리 화면이 하는 모든 기능 (CRM이 재현해야 할 목록) |
| `03_몰연동규약_provision.md` | DB 소유권 규칙 + 발급 후 몰 provision API 규약 + 봉인할 몰 경로 |
| `04_체크리스트.md` | CRM 구축 체크리스트 + 몰 변경 체크리스트 + 검증 |

## 권장 최종 구조 (옵션 D)
- **발급(계약·아이디·비번 생성) = CRM 단독**
- **운영(정지/활성/멘토 재배정/조회) = 몰 유지**
- CRM이 row 생성 후 몰 `provision` API 호출 → 몰이 기본 링크·고객그룹 등 부수효과 생성

## 확정 필요(회원님/CRM팀)
1. 계약서 작성 주체 = CRM (몰 approve 발급 제거 예정)
2. CRM이 User(mallUserId+비번)까지 직접 생성 가능 여부 → 가능하면 옵션 A까지 단순화 가능
