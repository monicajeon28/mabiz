
## 🔧 **구체적으로 뭘 바꾸는가?** (3가지)

### 1️⃣ **"정직 업그레이드"** (완료 vs 예정 명확히)

#### 현재 (나쁜 예)
```
메모리에 적힌 내용:
✅ SMS 자동화 완료 | 다음: Google Drive 백업 구현 예정

→ 뭘 완료했는지 몰라
→ 언제쯤 할 건지 몰라
→ 증거가 없어
```

#### 업그레이드 후 (좋은 예)
```
메모리에 적힌 내용:
✅ SMS 자동화 완료
   - Day 0-3 시퀀스 완성 (메시지 3개)
   - PASONA 심리학 적용 (이름/상황/제안/긴급/액션)
   - A/B 테스트 자동화 (메시지 2가지)
   - 커밋: a1b2c3d ← 이게 증거!
   - 파일: src/app/api/sms/send.ts

📋 다음: Google Drive 백업 (별도 이슈#45로 추적)
   - 일정: 2026-06-25 예상
   - 누가: Agent-Team-B
```

**왜 하는가?**
- 뭘 했는지 명확 → 버그 찾기 쉬움
- 증거(커밋) 있음 → 믿을 수 있음
- 기한 명시 → "언제쯤이야?" 질문 없음

---

### 2️⃣ **"검증 업그레이드"** (중요한 일은 2번 확인)

#### 예시: 계약서 검수

```
현재 (위험)
Agent-1: "계약서 완성했습니다!" 
→ 바로 배포
→ 나중에 법률 오류 발견 💥

업그레이드 후 (안전)
Agent-1: "계약서 첫번째 검수 완료"
Agent-2: "계약서 두번째 검수 (반대 입장에서 문제점 찾기)" 
→ Agent-1과 Agent-2 모두 "OK" 할 때만 배포 ✅
```

**언제 하는가?** (고위험 작업만)
- 고객과의 계약서
- 돈 계산하는 것 (수당, 수수료)
- 중요한 규칙 변경

**왜 하는가?**
- 실수 줄임
- 고객 피해 방지

---

### 3️⃣ **"쉬운 말 업그레이드"** (기술용어 제거)

#### 현재 (어려운 말)
```
UI에 이렇게 표시됨:
"L1 렌즈 감지 활성화"
"퍼널 자동화 시작"
"RBAC 권한분리"

→ 50대 파트너: "이게 뭐예요?" 😕
```

#### 업그레이드 후 (쉬운 말)
```
UI에 이렇게 표시됨:
"가격민감 고객 자동인식"
"자동메시지 시작"  
"역할별 권한설정"

→ 50대 파트너: "아, 이거구나!" ✅
```

**어떤 단어를 바꾸는가?**

| 어려운 말 | 쉬운 말 | 설명 |
|---------|--------|------|
| **L1 렌즈** | **가격민감 고객** | 가격이 비싸다고 투덜거리는 사람 |
| **L3 렌즈** | **경쟁사비교 고객** | 다른 회사와 비교하는 사람 |
| **퍼널** | **자동메시지** | 자동으로 보내지는 메시지들 |
| **워크플로우** | **일처리** | 일하는 과정/흐름 |
| **RBAC** | **역할별권한** | 매니저/판매원이 할 수 있는 일 구분 |
| **렌즈감지** | **고객특징파악** | 고객이 어떤 타입인지 알아내기 |

## ✅ **실제 해야 할 일** (작업 지시서)

### Phase 1️⃣ (1-2주) — 정직성 개선

#### 작업 #1: MEMORY.md 형식 개선
```
목표: 메모리 파일을 명확하게 구분

현재:
✅ SMS 완료 | 다음 작업 예정

변경 후:
✅ 완료 (증거: 커밋번호, 파일)
📋 작업중 (진행률, 예상날짜)
🗺️ 로드맵 (향후 계획)

누가: 엔지니어 1명
시간: 3시간
확인: 메모리 10개 파일 수정 완료
```

#### 작업 #2: 커밋 메시지 표준화
```
목표: 뭘 했는지 명확하게 기록

예시:
❌ 나쁜 예: "SMS 업데이트"
✅ 좋은 예: "feat: SMS Day 0-3 자동화 + PASONA 심리학 적용"
           - Day 0: 초기 메시지 (Problem 단계)
           - Day 1: 이의대응 (Solution 단계)  
           - Day 2: 가치강조 (Offer 단계)
           - Day 3: 긴박감 (Action 단계)
           - 테스트: A/B 자동실행 (50개 샘플)

누가: 모든 엔지니어
시간: 각 커밋 시 1분 추가
확인: 모든 새 커밋이 템플릿 따름
```

### Phase 2️⃣ (2-3주) — 검증 강화

#### 작업 #3: 계약서 이중검사
```
목표: 계약서 배포 전에 2명이 검수하기

과정:
1️⃣ Agent-검수팀-A: "계약서가 맞나요?" 
   - 법률 관점에서 검수
   
2️⃣ Agent-검수팀-B: "이 계약서의 문제점은?" 
   - 반대 입장에서 허점 찾기
   
3️⃣ 둘 다 "완벽해요!" 할 때만 배포 ✅

파일: src/app/api/contracts/[id]/sign/route.ts
누가: 엔지니어 1명 (Orchestration 구현)
시간: 1주일
확인: 계약서 배포 전 2번 검수 자동화
```

#### 작업 #4: 수당계산 정확성 테스트
```
목표: 수당 계산 99.9% 정확도 보장

과정:
1️⃣ 계산 완료 후 자동 검사
   - 합계가 맞나?
   - 소수점은?
   - 세금 계산은?
   
2️⃣ 샘플 데이터로 A/B 비교
   - 구 시스템 vs 신 시스템
   - 결과가 같으면 OK ✅

파일: src/app/api/affiliate/commission/calculate/route.ts
누가: 엔지니어 1명 (테스트 코드)
시간: 1주일
확인: 수당 계산 테스트 30개 통과
```

## 📊 **결과: 뭐가 좋아지는가?**

### Before (현재)
```
고객: "SMS 자동화가 뭐예요?"
→ 복잡한 설명 듣고 더 헷갈림

내부: "SMS 배포했는데 뭔가 이상해"
→ 뭘 했는지 기록이 없어서 찾기 어려움

법무팀: "계약서 맞나요?"
→ 검사 안 하고 바로 배포 → 나중에 문제 💥
```

### After (업그레이드 후)
```
고객: "자동메시지가 뭐예요?"
→ 도움말 클릭 → "아! 이거구나" ✅

내부: "SMS 배포했는데 뭔가 이상해"
→ 커밋에 명확히 기록됨 → 5분 안에 원인 찾음 ✅

법무팀: "계약서 맞나요?"
→ 2명 자동 검수 후 배포 → 안전 ✅

파트너: "이번달 수당 얼마예요?"
→ 99.9% 정확도 + 증거데이터 → 신뢰도 ⬆️⬆️
```
