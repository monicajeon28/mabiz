🎯 Cron Job 핵심 함수
────────────────────────────────────────────────────────────────────────────

1️⃣  getPendingExecutions(organizationId, limit?)
    ├─ 역할: PENDING 상태 메시지 조회
    ├─ 조건: status='PENDING' AND scheduledAt <= NOW()
    ├─ 인덱스: idx_execution_cron_scan (organizationId, status, scheduledAt)
    └─ 반환: ExecutionRecord[]

2️⃣  getExecutionDetailsWithContact(executions, organizationId)
    ├─ 역할: ExecutionLog 상세 조회
    ├─ 조인: Contact + Campaign 데이터
    └─ Phase 2: 실제 발송 데이터 병합

3️⃣  executeCampaignMessages(executions, channel)
    ├─ 역할: 메시지 발송 (stub in Phase 1)
    ├─ 채널: SMS | EMAIL
    ├─ 배치: 50명씩 처리
    └─ Phase 2: Aligo API, SMTP 구현

4️⃣  updateExecutionStatus(executionId, status, failureReason?, retryCount?)
    ├─ 상태: PENDING → SENT / FAILED / RETRY_SCHEDULED / ABANDONED
    ├─ 재시도: 지수 백오프 (1분, 2분, 4분)
    ├─ 최대: 3회 재시도
    └─ nextRetryAt 자동 계산

5️⃣  executePendingCampaigns() [메인 Cron Job]
    ├─ 모든 ACTIVE 조직 순회
    ├─ SMS/Email 채널별 병렬 처리
    ├─ 에러 핸들링 (조직별 격리)
    └─ 반환: { success, failed, duration }

3. 핵심 설계 원칙
================================================================================

✅ 중복 방지 (Idempotency)
   └─ UNIQUE(sourceType, sourceId, contactId, executeMonth)

✅ 재시도 로직 (Exponential Backoff)
   └─ 1분 → 2분 → 4분 → ABANDONED (max 3회)

✅ 배치 처리 (성능)
   └─ 50명씩 + 1초 딜레이 (API 레이트 리미팅)

✅ 채널 분리 (확장성)
   └─ SMS / Email 독립 처리 (향후 Push 추가 가능)

✅ 조직 격리 (안정성)
   └─ 한 조직 에러 ≠ 다른 조직 영향
