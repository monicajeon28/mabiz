# P2 사용자 경험(UX) 검증 체크리스트

## 개요

P2 최적화는 `/api/auth/me` 중복 호출 7개 페이지를 **미들웨어 기반 인증**으로 전환합니다.
- **기술 변화**: API 호출 제거 → 서버 헤더(`X-User-Role` 등) 기반 신뢰
- **사용자 경험**: 🎯 **변화 없어야 함** (로딩만 더 빨라야 함)
- **위험도**: 🔴 **HIGH** (인증 기반 페이지 변경, 다중 탭 시나리오)

---

## 1. 영향받는 사용자 상호작용

### 1.1 페이지별 상호작용 매트릭스

| 페이지 | 사용자 역할 | 상호작용 | 현재 방식 | P2 변경점 |
|------|---------|--------|--------|---------|
| **PNR 조회** (`/pnr/[id]`) | 공개 고객 / AGENT+ | 전화번호 검증 → 예약 조회 | 1. 폰 검증 2. `/api/auth/me` 호출 3. isAdminMode 판단 4. 데이터 로드 | 1. 폰 검증 2. isAdminMode 제거 3. 데이터 즉시 로드 (20-40ms 단축) |

| **제휴사 현황** (`/team/affiliate`) | OWNER+ | 제휴 담당자 조회, SMS 발송 | 3개 useEffect에서 각각 `/api/auth/me` 호출 | 1회 호출 제거 (150-300ms 단축) |

## 2. 각 페이지별 상세 검증 시나리오

### 2.1 PNR 조회 페이지 — `/pnr/[reservationId]` (⭐ 최고 위험도)

**현재 상황**: 
- 1,000회/일 중복 호출 (월 30,000회)
- 공개 고객(인증 불필요) + AGENT+ 관리자(인증 필수) 혼합
- `isAdminMode` 판단 로직이 `/api/auth/me` 응답에 의존

**P2 변경**:
- ❌ `/api/auth/me` 호출 완전 제거
- ❌ `isAdminMode` 상태 제거 (추론 불가)
- ✅ 공개 고객 경로는 유지 (전화번호 검증만)
- ❌ 관리자 모드는 별도 경로(`/dashboard/pnr/[id]`) 제공 예정

        ↓
┌─────────────────────────────────────────────┐

│ 예상: /api/pnr/customer/[id] 호출           │
│      응답: 예약 정보 + 여행자 목록           │

│ 액션: 여행자 수만큼 객실 번호 할당          │
│      "제출" 버튼 클릭                        │
│ 예상: /api/pnr/customer/[id] PATCH 요청   │
│      성공: 토스트 "객실 배정 완료"          │
│      실패: "객실 배정 오류" 토스트           │

⏱️ 예상 전체 시간: 이전 2,500ms → P2 1,800ms (28% 단축)

## 3. 수용 기준 (Acceptance Criteria)

### 3.1 무조건 같아야 할 것 (기능적 동일성)

```
✅ 로그인 상태 감지
  - 로그인 사용자 접근 → 페이지 로드 (동일)
  - 로그아웃 사용자 접근 → /sign-in 리다이렉트 (동일)
  
✅ 역할별 접근 제어 (더 엄격해야 함)
  - GLOBAL_ADMIN만 /admin/* 접근 가능
  - OWNER+만 /dashboard/team/* 접근 가능
  - 비로그인 사용자 접근 불가
  - 현재: 클라이언트 검증 → P2: 서버 검증 (더 안전)
  
✅ 에러 메시지
  - "전화번호 미일치" 메시지 동일
  - "권한이 없습니다" 메시지 동일
  - "서버 오류가 발생했습니다" 메시지 동일
  
✅ 폼 제출 & 데이터 변경
  - 신청서 승인 → 목록 즉시 갱신 (동일 또는 더 빠름)
  - 객실 배정 제출 → 토스트 메시지 표시 (동일)
  - 필터 변경 → 데이터 재로드 (동일)
  
✅ 다중 탭 세션 동기화
  - 탭 A에서 로그아웃 → 탭 B 새로고침 시 즉시 리다이렉트
  - 현재: 클라이언트에서 감지 (느림) → P2: 서버에서 감지 (빠름)
```

### 3.2 개선 가능한 것 (성능)

```
🟢 페이지 로딩 시간
  - 현재: 600-1,200ms
  - 목표: 450-950ms
  - 허용 범위: ±10% (예: 500ms → 450-550ms)
  
🟢 API 응답 시간
  - 현재: 150-250ms (auth/me 포함)
  - 목표: 100-200ms
  - 기준: authChecked 대기 시간 제거
  
🟢 서버 에러율
  - 현재: ~0.5% (일일 1,550회 중 일부)
  - 목표: < 0.3%
  - 감시: 401/403 에러 로그 모니터링
```

### 3.3 허용 불가능한 변화

```
❌ 사용자가 접근 불가능한 정보 노출
  예: 비로그인 사용자 → 정산금 정보 노출 (보안 실패)
  
❌ 로그인 상태 미감지
  예: 로그아웃 후 페이지 새로고침 → 낡은 데이터 표시 (Hydration 오류)
  
❌ UI 깜박임 (Flash)
  예: 페이지 로드 → 권한 없음 → 리다이렉트 (UX 나쁨)
  → 서버에서 미리 검증하면 해결됨
  
❌ 예상 못한 리다이렉트
  예: 사용자가 신청서 검토 중 → 갑자기 홈으로 리다이렉트
  → 세션 만료 시에만 리다이렉트 (명확한 사유)
```
